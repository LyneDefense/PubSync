# Database migrations (Alembic)

Alembic is wired to the application: `env.py` uses `Base.metadata` as the target
and reads `DATABASE_URL` from app settings, so commands run against the configured
database with a single source of truth.

## Status / intent

This was added to collapse the hand-rolled schema bootstrap (`create_db_and_tables`
+ `ensure_runtime_schema` in `app/database.py`) into versioned migrations.

- `0001_baseline` reproduces the **current full schema** by delegating to the app
  bootstrap. It is the starting point.
- `ensure_runtime_schema()` is still called on app startup as a **parallel
  fallback** — adopting Alembic is additive and does not remove existing behaviour.
  Once you have verified Alembic end-to-end you can switch startup to run
  `alembic upgrade head` instead and retire the runtime bootstrap.

## Commands (run from `backend/`, with the venv active)

Bring a fresh database up to the latest schema:

```bash
alembic upgrade head
```

If your database already matches the baseline schema (the common case for the
existing dev DB), stamp it instead of re-running:

```bash
alembic stamp 0001_baseline
```

Create a new migration after changing the ORM models in `app/models.py`:

```bash
alembic revision --autogenerate -m "describe the change"
alembic upgrade head
```

Inspect state:

```bash
alembic history
alembic current
```

> Note: `0001_baseline` opens a real connection (it calls the app bootstrap), so it
> must run in online mode. Offline `--sql` generation is not supported for it;
> autogenerated future revisions support `--sql` normally.
