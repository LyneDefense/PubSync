# 成长趋势 + 受众需求 优化方案设计(v1 定稿)

> 起草 2026-07-06 · 状态:已评审,准备实施(小红书优先)
> 涉及:博主画像页「成长趋势」重做 +「爆款归因」退役换成「受众需求」。

---

## 0. 一句话

把「一张看不懂的轨迹图 + 一段不准的 LLM 归因」换成:**两张能读的小图(基线走势 / 表现分布)+ 从评论里挖的受众需求(读者最常问)**。原则:**便宜的数据事实常驻;可复制的"打法"全部交给蒸馏,分析层绝不重做;小红书拿不到的(推流/热点)诚实不装。**

---

## 1. 背景

对标学习工具,用户要的是「TA 身上什么可复制、我该抄的」。现有两模块没答好:成长趋势「太零碎看不出东西」,爆款归因「不准」。

关键边界(评审确定):**"可复制的打法" = 蒸馏(创作画像)的活**,它已经产出真的标题公式/开头模板/选题思路/口播结构/语言 DNA,而且建档时就把"最新 N + 历史高赞 M"喂进去专门学爆文。**分析层不再造一个更粗的平行版本**,只留蒸馏产不出、且正交的事实。

---

## 2. 诊断(基于代码)

### 2.1 成长趋势 `trajectory.py` + `DossierTrajectory.vue`
1. 每篇一个赞数散点,赞跨 2–3 个数量级、受选题运气/推流驱动 → 噪声云。
2. **线性 Y 轴**把 90% 笔记压成底部一条线,一个爆文撑开整轴 → 只见刺不见势。
3. 一张 236px 的图叠了 7 层(散点+基线+面积+阶段带+网格+拐点虚线+爆发点+轴),无焦点。
4. 阶段划分"有机器没回报";涨粉拐点是从"赞基线跳高"**猜**的,不是真涨粉数据。

### 2.2 爆款归因 `attribution.py`
1. **对照组只送标题+赞**(`_render_normals`),爆文送完整内容 → LLM 没法真对比 → 踩进幸存者偏差。
2. 无量化骨架,产出不可证伪的散文 → "不准"。
3. **因果框架本身不可靠**:推流/热点/运气不可观测。

---

## 3. 设计原则

1. **便宜的事实常驻**:趋势、分布 compute-on-read、$0、常显。
2. **打法归蒸馏**:分析层不复述"怎么写",要看打法就指向创作画像。
3. **正交才留**:只做蒸馏产不出的——账号时序表现、分布、**受众侧**信号(评论)。
4. **诚实**:赞≠粉(互动代理);小红书无浏览量 → 推流不装;热点回溯拿不到 → 不装。

---

## 4. 方案 v1

### 4.1 成长趋势 → 拆成两张小图
去掉阶段带、涨粉拐点虚线等杂线。**"涨粉拐点"概念退役**(赞数拿不到涨粉量)。

- **图 A · 互动基线走势**:Y 轴取对数(或按全局中位归一);主角 = **按月/季分桶的互动中位基线线 + P25–P75 分位带**(阴影);散点默认去掉/极弱;右上角**近期走势徽章**(走强/平稳/走弱);最多标 1 个最高爆文;底部轴说明 + 一句阶段文字(不画带)。
- **图 B · 表现分布**:**log 直方图**,标 p25 / 中位 / p75 / p90 + 头部落点 → "大部分笔记 X–Y 赞,头部到 Z",区分"稳定腰部"型 vs "偶发爆款"型。

后端(`trajectory.py`,compute-on-read,无迁移):bucket 增 `p25/p75`;顶层增 `distribution={count,min,p25,median,p75,p90,max,log_bins[]}`;phases/level_ups 仍算但前端只当文字、不画带/虚线。

### 4.2 受众需求 · 读者最常问(替代爆款归因)
从 top 详情笔记的**读者评论**(`is_author=false`)聚合,LLM → **「读者最常问的 N 件事 / 高频关注点」**。这是选题金矿,从**受众侧**来,与蒸馏(创作者侧)正交,且不涉因果。

**v1 用现有评论,零新增采集**:现状每篇详情采**至多 20 条顶层评论、按点赞热度排序**(`get_note_comments` 的 `sort_strategy=like_count`)——拿到的天然就是"最多人问"的那批。几十篇 × 20 ≈ 大几百上千条,够 LLM 提主题。

- LLM、按钮触发(省钱,同现在归因的触发方式);结果存库(复用 `attribution_json` 列,归因退役后该列腾出来,**无迁移**),读时展示。
- 输出:`{questions: [{theme, ask_count_hint, sample}], focus_points, note}`(note 说明依据 M 篇评论、非穷举)。

### 4.3 轻标注:可复制 → 指向蒸馏
成长趋势里对头部爆文给一句轻标注的口子:若某爆文与其它高表现笔记有共性 → "可复制,详见创作画像";孤峰无共性 → "偏一次性,别硬抄"。**只指路,不复述打法**。(v1 可先只放"去看创作画像"的指引,共性判断放宽到后续。)

### 4.4 爆款归因退役 · 推流/热点(小红书)诚实不做
- 归因整块退役。
- **小红书无浏览量** → 推流强度是黑盒,不推断、不假装。
- **热点回溯拿不到**(无历史热榜库) → 不做;仅在措辞上允许"时效型选题,不一定可复制"的弱提醒。

---

## 5. 数据与接口

**后端**(无迁移):
- `trajectory.py`:加 p25/p75 + distribution。
- `audience.py`(新):抽读者评论 → LLM 读者最常问。
- `attribution.py`:退役/改为 audience;`service.dossier_overview` 用 `audience` 替 `attribution`,趋势带 distribution;归因端点改跑 audience(复用 `attribution_json` 存储)。
- `schemas/blogger.py`:trajectory 加 p25/p75/distribution;新 `DossierAudience`;去 `DossierAttribution`(或改名)。
- 单测:分位/distribution、log 分箱、读者评论过滤(is_author)、audience 解析兜底。

**前端**:
- `DossierTrajectory.vue`:重写为两张小图。
- `DossierAudience.vue`(新):读者最常问 + 按钮触发。
- `BloggerDossierView.vue`:账号事实带 = 数据面板 → 成长趋势(两图)→ 受众需求 → 运营习惯|合规。
- `api/types.ts` + `useDossier.ts`:新类型;`distribution` 随 overview;audience 走按钮流。

---

## 6. Phase 2(验证 v1 有价值后再上)

- **楼中楼(评论下评论)**:解锁"读者追问 + 博主回答"的对话;但每条有回复的评论要多一次调用,**很贵**,且与「运营习惯·博主回复」重叠 → 延后。
- **多采评论**(20→30/50):对评论多的爆文抓长尾问题,费翻页钱 → 按需调。
- **抖音推流**:抖音有播放量,可用"赞/播比"粗判推流/投流——本轮小红书优先,不做。
- **热点数据源**:接外部热榜(新榜/微热点),只对往后实时有用。

---

## 7. 诚实边界(写进 UI 文案)

- 赞 ≠ 粉:趋势是**互动**代理,拿不到官方涨粉曲线。
- 小红书无浏览量:推流/曝光看不到,不装。
- 受众需求依据**已采顶层评论**,非全量、非穷举。
- 打法看创作画像,这里不重复。
