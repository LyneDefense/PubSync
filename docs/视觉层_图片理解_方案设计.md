# 视觉层（图片理解）方案设计

> 目标：给"图文优先"的小红书补上缺失的另一半内容通道——图片里的文字与视觉表达。
> 本方案与已上线的 ASR 层**结构对称**：ASR 解决"内容在视频语音里"，视觉层解决"内容在图片里"。
> 文档含：架构 / 数据模型 / 采集接入 / **下游五大功能的补强与实现** / 配置 / 成本 / 分期 / 边界 / 关键文件 / 验证。

---

## 一、Context（为什么做）

小红书是**图片优先**的平台。今天系统对图文笔记是**半瞎的**：只读 `title + body_text + 评论`，而真正的内容常常全在图里：

- **封面**＝钩子/标题党（"每天拆解一个博主""1个月涨粉2万"），决定点不点进来；
- **图内文字**＝真干货：拆解卡片要点、教程步骤、数据截图、求租房的详细要求、清单、对比表；
- **视觉风格**＝版式/配色/字体/信息密度，是博主人设与辨识度的一部分。

这跟刚上线的视频 ASR 是同一类问题的两半：

```
视频笔记：内容在「说的话」里 → ASR 提取（transcript_text）        ✅ 已上线
图文笔记：内容在「图片」里   → VLM 提取（image_text + visual_digest） ← 本方案
```

同时能顺手修掉两个现象：模态分类偶发"未知"（图文有真实字数后密度判定更准）、以及"无音频流 h264 封面"被跳过（那本就是图片封面，VLM 正好能读）。

## 二、目标与非目标

**目标**
- 为图文笔记（及可选的视频封面）产出 `image_text`（图内逐字 OCR）与 `visual_digest`（封面话术/版式/风格/信息点的结构化摘要），入库、可复用、可补解析。
- 让**采集、蒸馏、Skill 优化、对标分析、诊断我的**这五个功能"自动"读到图片内容——通过一个共享的"笔记正文装配"入口，一处接入、处处受益。
- 后台全局开关 + 模型档位 + 每篇图数上限，成本可控、可降级、不掀翻采集。

**非目标（本轮不做）**
- 不做"生成图片"（那是 CogView 的活，见 §7.3 说明如何衔接）。
- 不硬存原图（只存已有的图片 URL + 提取出的文字/摘要）。
- 不做跨笔记的图片去重/相似度聚类（后续增强）。

## 三、总体架构：第二个"传感器" + 一处装配

### 3.1 视觉模型（GLM-4.6V）
- 用智谱 **GLM-4.6V**（128k、强 OCR/文档解析，~1元/M 输入·3元/M 输出；后台可切 4.5V/4v-plus）。
- **OpenAI 兼容 `/chat/completions`**，图片走 `{"type":"image_url","image_url":{"url":"…"}}`，**可直接传公网 URL**（我们的图片 URL 已存在 `cover_url` / `media_urls_json`），也支持 base64；单次最多 5 张图。
- 复用现有 GLM 通道（`app/services/ai_service.py` 的 `glm_*`），凭据同文本/配图/ASR（在后台加密库里，不在 .env）。

### 3.2 一处装配（关键设计）
今天没有统一的"笔记→文本"入口，各功能各自拼 `body_text + transcript_text`（见 `analysis.py:341`、`rollout.py:31`、`account_audit/service.py:145`）。本方案新增：

```
app/blogger_distillation/post_content.py
  assemble_learnable_text(post) -> str      # 标题+正文+口播稿+图内文字，一篇笔记的完整"可学文本"
  visual_context(post) -> str               # 从 visual_digest 拼出「封面话术/版式/风格/信息点」紧凑块
```

把上述三处消费点改成调用这两个函数——**这就是"一处接入、五个功能受益"的落点**。以后再加新模态（如视频关键帧文字）也只改这一个文件。

## 四、数据模型

`BloggerPost` 新增列（与 ASR 那组 `transcript_text/asr_status/asr_error` 对称）：

| 列 | 类型 | 说明 |
|---|---|---|
| `image_text` | Text default "" | 图内逐字 OCR 拼接（封面在前，正文图按序在后） |
| `visual_digest` | Text default ""（存 JSON 字符串） | 结构化视觉摘要：`{cover_hook, layout, style, info_points[]}` |
| `vision_status` | String(30) default "not_required" | `not_required/pending/succeeded/skipped/failed`（对齐 `asr_status`） |
| `vision_error` | Text default "" | 失败/跳过原因 |
| `vision_image_count` | Int default 0 | 实际解析了几张图（成本与覆盖度可观测） |

- `cover_url`、`media_urls_json` **已存在**，Phase 1（封面）无需新采数据。
- 迁移 **`0018_post_vision_layer.py`**（最新是 0017）：幂等加列 + `server_default`，沿用 0017 的 `_columns()` 判断写法。

## 五、视觉解析模块（镜像 ASR）

```
app/blogger_distillation/vision.py         # 镜像 asr.py
  class VisionResult:  image_text, visual_digest(dict), image_count, provider
  build_vision_provider(settings) -> VisionProvider | DisabledVisionProvider
  class GlmVisionProvider:
      analyze_images(image_urls, *, source_id, on_progress) -> VisionResult
        - 取前 N 张（Phase1: 仅 cover；Phase2: cover + 正文图前 N，N=vision_max_images_per_note）
        - 组 chat/completions：一条 user message，content=[文本指令, image_url×≤5]
        - 解析返回 JSON；失败重试（4xx 快失败、5xx/网络重试，复用 ASR 的重试范式）
        - 图 URL 拉取失败（防盗链）→ 下载后 base64 兜底（复用 asr.download_file）
```

**Prompt（两件事，产出 JSON）**
1. **逐字 OCR**：转写图中所有可读文字，保持原文，看不清标注 `[模糊]`，**绝不编造或补全**；
2. **视觉摘要**：`cover_hook`（封面主文案/钩子）、`layout`（版式：卡片/清单/对比/大字/截图…）、`style`（配色/字体/信息密度/风格调性）、`info_points`（图里传达的关键信息点，≤6条）。

```
app/blogger_distillation/service/vision_step.py   # 镜像 asr_step.py
  handle_note_vision(db, tenant, task, candidate, normalized, vision_provider, blogger):
    - 复用：按 note_key 命中已成功解析则跳过（省成本，对齐 _reuse_existing_transcript）
    - provider 为 None（关闭/初始化失败）→ vision_status=skipped，记事件，返回
    - 选图 → analyze_images → 写 normalized[image_text/visual_digest/vision_status/vision_image_count]
    - 事件："视觉解析" running/succeeded/failed，多图时回报「解析中…第 x/n 张」
```

## 六、采集流水线接入（`service/collection.py`）

1. `collect_posts` 里像 `asr_provider` 一样构建一次 `vision_provider`（`settings.vision_enabled` 时），记录"视觉解析已开启：model=…"事件。
2. 主循环内，对**有图的笔记**（图文一定有；视频取封面）调用 `handle_note_vision`（放在 ASR 之后、评论之前）。
3. **增量补解析**（镜像现有"ASR 补转写"，见 `collection.py:201`）：增量分流时，已存在的笔记若 `vision_enabled` 且 `vision_status ∈ {skipped, failed, ""(存量)}` → 加入 `to_fetch` 重抓详情 + 视觉解析（封面/图片 URL 带签名会过期，故需重抓详情拿新 URL）。
4. 事件文案与 ASR 一致，前端 `LiveProgress` 自动展示，无需前端改动即可看到"视觉解析"进度。

> 存量回填：老笔记 `vision_status` 默认空，再采一次即被补解析（和 ASR 补转写同机制）；也可像当初 note_key 那样写一次性回填脚本。

## 七、下游五大功能的补强与实现

> 统一原理：把各功能的"取笔记文本"改为 `assemble_learnable_text(post)`，把"视觉上下文"用 `visual_context(post)` 追加进 prompt。下面逐个给**现状 → 加视觉后 → 实现落点**。

### 7.1 采集（Collection）
- **现状**：图文只存 标题/正文/评论；模态偶发"未知"；封面/无声视频空手。
- **加视觉后**：图文带 `image_text + visual_digest`；模态密度判定有真实字数、更准；封面被读取。
- **实现**：§四（列）+ §五（模块）+ §六（接入）；`modality.classify_subtype` 增加 `image_text` 作为文本信号（`collection.py:579` 传入），把"未知"降到更少；可选新增图文子车道判定（干货卡片 vs 生活方式，按 `visual_digest.layout`）。`schemas/blogger.py::BloggerPostRead` 增 `image_text/visual_digest/vision_status`；前端笔记详情/抽屉展示"图内文字/封面解析"（只读）。

### 7.2 蒸馏（Distillation）——最大赢家
- **现状**：从单薄文字学；`analysis.post_summary`（`analysis.py:350`）喂给蒸馏的每篇只有 `body_excerpt/transcript_excerpt`。
- **加视觉后**：蒸馏看到选题、信息密度、**封面话术公式**、视觉人设；图文车道终于有真货；主张有图内实证可 grounding。
- **实现**：
  - `analysis.post_summary` 增 `image_text_excerpt`、`visual_digest`、`has_image_text`；`extract_opinion_sentences`（`analysis.py:341`）的 `source_text` 并入 `image_text`。
  - `distill_engine` 的内核 prompt：**人设/风格**维度显式纳入 `visual_context`（版式/配色/信息密度）；**图文车道** prompt 增"封面公式/图内信息编排"抽取维度，产出的 Skill 会写"图片怎么做"，不只"文字怎么写"。
  - grounding（下一层）：引用 `image_text` 具体片段做证据。

### 7.3 Skill 优化（SkillOpt）
- **现状**：`rollout.gold_text`（`rollout.py:31`）图文模态只用 `title+body` 当风格参照；StyleDist 只对齐文字风格。
- **加视觉后**：风格对齐扩到**图文表达**（封面话术/版式）；优化目标从"像不像本人的字"→"像不像本人的图文"。
- **实现**：
  - `rollout.gold_text` 图文模态改为 `assemble_learnable_text(post)`（含 image_text）；`extract_topic` 同步。SkillOpt 从 DB 时间切分取样（M1-3）本就读 `BloggerPost`，新列自动进入。
  - 进阶：`build_generation_prompt` 图文分支要求产出 **封面主文案 + 版式建议**（info_points 的编排），再交 **CogView** 落地成图——这就是"优化图片"的闭环衔接点（本轮先让 Skill/创作产出图片指令，配图生成沿用已有 image_provider=glm）。

### 7.4 对标分析（Benchmark Discovery / Analysis）
- **现状**：候选评分与报告基于文字，看不到对标号真正强在哪。
- **加视觉后**：看得到对标号的**封面公式、信息卡套路、视觉辨识度**——常是他爆的真因；"可对标性/可迁移性"判断更准。
- **实现**：`benchmark_discovery` 引擎里"每条候选笔记文本"改用 `assemble_learnable_text + visual_context`；评分维度里"可迁移度/可对标性"的证据纳入视觉摘要；对标报告新增"封面公式/视觉打法"小节。注意：候选笔记多来自列表预览，需对候选样本也跑一遍视觉解析（受 `vision_max_images_per_note` 与候选数约束成本）。

### 7.5 诊断我的账号 / 对标诊断（Account Audit）
- **现状**：`account_audit/service.py:145` 组的每篇只有 标题/正文/口播；只诊断文案。
- **加视觉后**：能诊断"封面抓不抓人、图内信息清不清、视觉风格统不统一"（图文号核心体检项）；对标诊断能做**封面/版式差距**对比，而非只有文案差距。
- **实现**：`account_audit/service.py` 的 block 生成器每篇追加 `图内文字:{image_text[:400]}` 与 `封面:{visual_digest.cover_hook}`；体检 agent 的维度/prompt 增"封面吸引力、图内信息清晰度、视觉风格一致性"；对标诊断（我 vs 对标）增"封面公式差距、版式差距"对比项。

## 八、配置（config + admin/config_registry）

| 键 | 默认 | 说明 |
|---|---|---|
| `vision_enabled` | `true` | 全局开关（后台控制，用户端不暴露，对齐 asr_enabled） |
| `vision_provider` | `glm` | 供应商 |
| `vision_model` | `glm-4.6v` | 视觉模型（后台可切 glm-4.5v / glm-4v-plus-0111） |
| `vision_scope` | `cover`（Phase1）→ `cover_body`（Phase2） | 解析范围 |
| `vision_max_images_per_note` | `4` | 正文图上限（+封面），控成本 |
| `vision_download_fallback` | `true` | URL 拉取失败时下载转 base64 |

Registry 增一组 `vision`（或并入 `model`），`vision_*` 加入 `OVERRIDABLE`。

## 九、成本

- 一张图 ≈ 2k tokens ≈ **0.004 元**（GLM-4.6V）。
- Phase 1（封面全量，~300 篇）≈ **~1.2 元**；Phase 2（封面+4张正文图 × 300 篇）≈ **~6 元**。与 GLM-ASR 同量级，可忽略。
- 接入现有 `app/cost` 记账（GLM chat 调用即记）；成本预估接口 `collect-estimate` 增"预计视觉解析张数×单价"。

## 十、分期

- **Phase 1（先做，ROI 最高）**：仅**封面**。列 + 模块 + 采集接入 + 五功能读到 `image_text/visual_digest`（封面维度）。封面公式对蒸馏/对标/诊断提升最大。
- **Phase 2**：图文**正文图**（拆解卡片/教程/清单/求租房细节），每篇限 N 张。
- **Phase 3（可选）**：非口播视频的**关键帧**（内容在屏幕字幕而非口播），与 ASR 互补（复用 vision.py，抽帧后当图处理）。

## 十一、边界 / 风险

- **防盗链**：先直传图 URL；GLM 拉取失败 → 下载后 base64（复用 `asr.download_file`）。**上线前先真机验证一张 XHS 图 URL 能否被 GLM 直接拉取**。
- **OCR 幻觉**：prompt 强约束"逐字、看不清标 `[模糊]`、不编造"；`visual_digest` 只描述所见。
- **隐私**：只提取文字与风格摘要，不落地原图；图中若含联系方式/人脸——只 OCR 文字（博主自公开内容），不做人物身份识别；沿用评论匿名化基线，结果不外发。
- **降级不掀翻**：解析失败按 skipped/failed 记录并降级分析（镜像 `is_expected_asr_skip`），绝不让单篇失败掀翻整批采集。
- **成本护栏**：每篇图数上限 + 后台开关 + 复用 note_key。

## 十四、不做的事（本轮）

- 不生成图片（只产出封面/版式**指令**，落地交 CogView）。
- 不做图片相似度/去重聚类、不做逐帧视频视觉（Phase 3 再说）。
- 不落地存原图；不做人物身份识别。
- 不在用户端暴露视觉开关（后台全局控制，同 ASR）。
