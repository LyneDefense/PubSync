# 采集模块设计

> 博主内容采集(小红书 / 抖音)的设计定稿。目标:把采集从「每次全量重抓」重构成
> **「维护博主笔记池」**——增量更新、可选选材、ASR 可补、下架可对账、URL 兜底;
> 贵的操作(详情 / 评论 / ASR)只花在真正需要的笔记上。

## 1. 背景与问题

现状(重构前)每次采集都「按主页顺序拿前 N 条 → 全部重抓详情 / 重跑 ASR」,三个硬伤:

1. **重复浪费**:已采过的笔记又抓一遍详情、视频又重转一次 ASR。
2. **选材单一**:只能「最新」,选不了历史高赞,导致蒸馏偏最近内容。
3. **ASR 一次性**:忘了开 ASR → 视频全成「已跳过」、模态判为「未知」,补救只能整批重采。

## 2. 核心概念

- **笔记池**:某博主名下去重后的全部 `BloggerPost`(按 `external_id` 唯一)。反复采集只会
  更新 / 新增,不产生重复行。
- **采集批次 `BloggerCollectionRun`**:一次采集的成员快照(通过 `BloggerCollectionPost` 指向笔记行)。
  批次级统计冻结,单篇笔记内容取最新(被后续采集覆盖刷新)。
- **增量**:采集的唯一行为 —— 加新笔记 + 刷新老笔记互动 + 给跳过/失败的视频补 ASR。

## 3. 用户可见配置

发起采集时:

| 配置 | 取值 | 说明 |
| --- | --- | --- |
| 排序 `order` | `top_liked`(默认) / `latest` | 候选池怎么排:高赞优先 / 最新优先 |
| 数量 `amount` | `N 条` / `全部(到系统上限)` | 取多少 |
| 拉取范围 `content_types` | 图文 / 视频 | 不要视频可省 ASR 开销 |
| 每条评论数 `comments_per_post` | 0–100 | |
| 视频转写 `asr_enabled` | 开 / 关 | 只对被拉取的视频生效 |

**没有「采集模式」选择**:采集恒为增量。

- 不做「全量重采」:博主改了已发笔记正文是唯一边缘场景,用 §6「URL 定向采集」补那一条。
- 不做「仅刷新互动」独立模式:增量本就刷新老笔记互动。

## 4. 系统决定(用户端不暴露,`config.py` + 后台可调)

- **高赞候选池深度** = `clamp(sample_limit × 5, 100, 300)`。翻列表直到「攒够该深度 / 列表翻到底 /
  触达 300 上限 / 分页出错」任一为止;`amount=全部` 时用上限或翻到底。
- 常量:`candidate_oversample=5`、`candidate_floor=100`、`candidate_cap=300`。

## 5. 采集流水线(恒增量)

1. **拉资料 + 拉列表(便宜)**:`get_user_notes` 按候选池深度翻页,返回候选池 + `reached_end`
   (`has_more=false`);列表自带 `note_type` + 赞藏数。
2. **选材 → 目标集**:按 `order` 排序 → 按 `amount` 取数 → 按 `content_types` 过滤 → 一组 `external_id`。
3. **增量 diff**(目标集 vs 库内已有):
   - 新笔记 → 详情队列(贵:详情 + 评论 + ASR);
   - 老笔记 → 用列表赞藏数轻量刷新(免费,不抓详情);
   - ASR 补转写:老视频 `asr_status ∈ {skipped, failed}` 且本次 `asr_enabled` → 仅这几条重抓详情 + ASR
     (视频 URL 易过期,故需重抓该条详情)。
4. **详情采集(仅新 / 补 ASR)**:详情 → 规范化 → 空内容跳过 → 字幕/ASR → 评论 → 质量校验 → `upsert_post`。
5. **打 `content_subtype`**:按 `content_type` + 转写密度(图文 / 口播视频 / 非口播视频 / 未知)。
6. **更新 `last_seen_at`**:本次列表出现过的(新 + 老)post 都更新为现在。
7. **下架对账(仅当 `reached_end=true`)**:库内该博主、`status=active` 但本次完整目录未出现的 post →
   标 `status=delisted`。**部分采集绝不标**(避免误杀)。
8. **建批次成员 + 统计**:`BloggerCollectionPost` + `analyze_posts`(互动按模态归一)+ 自动标签 →
   批次 summary + `new_count` / `refreshed_count`。

### 反复采集的代价(示例)

已采 20 条高赞;第二次仍「高赞优先 + 20 条」:
- 翻 100 条列表(便宜)→ 排序取 top20 → diff;
- 没有新爆款 → 命中 0 条新 → **不抓任何详情 / ASR**,仅免费刷新数据;
- 有 X 篇新爆款挤进 top20 → 只抓那 X 篇;掉出的老笔记留池不动。
- **详情 / 评论 / ASR 对已采笔记绝不重抓**(`external_id` 去重 + 增量跳过)。

## 6. 兜底:粘贴 URL 定向采集

列表采不到想要的笔记时,允许粘贴笔记链接精确补采。
`get_image_note_detail` 仅需 `note_id + xsec_token`,均可从分享链接解析。

- **流程**:粘贴一行/多行链接 → 解析每条(从 `discovery/item|explore/{note_id}?xsec_token=...` 取 id+token;
  短链 `xhslink.com` 先跟一次重定向展开)→ 构造 candidate → 详情 + 评论 + ASR →
  **校验作者 == 当前博主** → `upsert_post` 进笔记池。
- **前端提示**:「打开小红书 → 进入该笔记 →「分享」→「复制链接」→ 粘到下面(一行一个);
  链接需带 `xsec_token`(复制分享链接即可),短链会自动展开。」
- **失败回报**:逐条标注 成功 / 失败原因(链接无法解析、缺 token、笔记已删除或不可见、
  作者与所选博主不符、平台返回错误)。
- 平台:小红书 + 抖音(抖音分享链接 → aweme_id → video_detail)。

## 7. 数据模型变动

`BloggerPost` 新增:
- `last_seen_at: datetime | None` —— 列表出现即更新,下架对账用。
- `status: str = "active"` —— `active` / `delisted`,下架默认不进蒸馏、历史保留。

`BloggerCollectionRun`:记录 `order` / `amount` / `new_count` / `refreshed_count`(透明)。

迁移 `0008_post_lifecycle.py`:幂等加列(沿用 `_columns()` 判断 + `server_default`)。
`content_subtype` 已在 0007 上线,无需再动。

## 8. 成本

- 贵的(详情 / 评论 / ASR)只发生在:新笔记 + ASR 补转写 + URL 定向采集的那几条;老笔记走免费列表刷新。
- **成本预估按「预计新增 N 篇」**(diff 后)算,不按 `sample_limit`。

## 9. 边界 / 异常

- 详情失败 / 空内容 / 受限 / 质量不合格 → 跳过并记事件。
- 分页中断 → 保留已拿到的候选;`reached_end=false` → 本次不做下架对账。
- 大号(> 300 篇)→ 高赞只在最近 ~300 篇里挑(记日志);永远翻不到底 → 不做下架对账。

## 10. 前端要求

- 采集向导第 2 步:加「排序 / 数量」选择,延续现有 `creator-steps` / `config-grid` /
  胶囊勾选样式与尺寸,不引入新风格。
- 「粘贴链接定向采集」:作为采集页一个折叠/次级入口,带提示文案 + 逐条结果列表(成功/失败原因)。
- 样本列表:`delisted` 的笔记加「已下架」中性标签(复用 `StatusChip` / tag chip 风格)。
- 成本预估文案改为「预计新增 N 篇」。
- 一切沿用 `assets/main.css` 既有 token(`--space-*` / `--radius-pill` / `.subtype-option` 等)。

## 11. 不做(本轮)

- 蒸馏侧「笔记池选材 / 命名快照 / 自动补」—— 蒸馏单独定。
- 视觉模型判图文相关性;公众号纯文章 / 配图文章(仅枚举留口子)。
- 硬删笔记(只标 `delisted`)。
