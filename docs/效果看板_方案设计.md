# 效果看板（数据看板）方案设计

> 状态：**方案定稿，待拍板范围后实现**。本文件只定"效果看板"，不动其它模块逻辑。
> 关联：[[Skill优化_方案设计]]、[[采集模块设计]]、[[商业模式与定位]]。

## 一、这个看板要回答什么

一句话：**「我用了 PubSync，到底变好了没、省了多少事？」**

不是堆图表，而是用我们手里**真实有的数据**去证明价值。证明不了因果的地方（涨粉），就**诚实地用相关性呈现**，不假装能证明。

## 二、核心结构：两层

采集/蒸馏是"对着对标博主做的准备工作"，创作/发布/涨粉才是"我某个账号的成果"。硬塞进单一账号会失真，所以分两层：

- **A 层 · 工作台总览（不分账号）**——"我用了多少工具"
  采集 N 次、蒸馏 N 次、AI 创作 N 篇、Skill 优化 N 次、对标库规模；每类**平均耗时**；最近活动时间线。
- **B 层 · 账号成果（按"我的账号"切换）**——"帮到了这个账号什么"
  涨粉趋势、为该账号创作/发布数与转化率、该账号对标博主的相似度趋近、产能与省时。

> 一个用户可有多个"我的账号"（`BloggerProfile.account_type='mine'`）。B 层顶部切换账号；A 层是全租户合计。

## 三、数据来源盘点（决定可行性）

### 已有，直接能用
- `OperationTask`（`task_type` + `created_at`/`updated_at` + `status`）：各类活动的**次数**与**平均耗时**（succeeded 任务 `updated_at − created_at`）。
  - 相关 `task_type`：`blogger_collection`（采集）、`blogger_distillation`（蒸馏）、`xhs_package_draft`（AI 创作）、`skill_optimization`（Skill 优化）、`account_audit`（体检）、`benchmark_recommend`（找对标）。
- `XhsPublishPackage`：AI 创作产出的草稿/作品数（`status` 默认 `generated`）。
- `SkillTrainingRun`：风格相似度 before/after、floor/ceiling 锚点、`gap_closed%` → 相似度趋近曲线。
- `BloggerPost`：每篇笔记的 `like_count`/`comment_count`/`share_count` + `published_at` → 篇均互动、爆款率、按发布时间的互动走势。
- 合规扫描结果（创作侧已有）：合规通过率。
- `CostEvent`：成本——**按项目约定只给后台 admin 看，用户端效果看板不展示金额**（见任务 #57）。

### 要补的数据关系（两处）
1. **创作绑账号**：`XhsPublishPackage` 加 `my_account_id`（创作时用户选目标账号）+ `published_at`（"我已发布"反馈）。
   → 没有它，创作/发布/省时无法按账号分。
2. **账号指标时序**：新表 `account_metric_snapshots` + **每日定时快照**。
   → 没有它，涨粉曲线和"活跃期 vs 沉默期"对比都画不出来（`BloggerProfile.follower_count` 只存当前值，每采集一次就覆盖）。

## 三·五、用户没绑"我的账号"时怎么办（降级原则）

很多用户初期只搞了对标博主、还没添加自己的账号。**绝不能因此让看板空白或报错。** 原则：

- **A 层永远能看**：采集/蒸馏/创作/优化的次数、耗时、对标库、产能省时、相似度趋近——全是租户级聚合，**和有没有"我的账号"无关**。所以看板首屏总有内容。
- **创作绑账号是"可选"，不是"必填"**：
  - 有账号 → 创作时下拉选目标账号（默认选第一个，可选"暂不指定"）。
  - **零账号 → 该选项隐藏/置灰**，提示"还没有自己的账号？可到『我的账号』添加，添加后能按账号看成果"。创作照常进行，`my_account_id` 存 `null`。
  - `my_account_id` 因此**可空**；未归属的创作只进 A 层合计，不影响 B 层。
- **B 层（分账号成果 + 涨粉）是"加分项"**：
  - 零账号 → B 层显示**引导空状态**："添加你的账号，解锁涨粉趋势、发布转化、分账号省时。" + 一个跳「我的账号」的 CTA，**不显示空图**。
  - 有账号 → 顶部账号切换器；只列真实"我的账号"（未归属创作不单列成账号）。
- **每日快照**：没有 `account_type='mine'` 的账号时，定时任务**什么都不做**（空转，不报错）。
- **结论式总结条**：按手头数据降级——零账号时只说工作台/产能省时/相似度那几句，不硬塞涨粉。

## 四、模块逐条设计

### 模块 1 · 工作台总览（A 层）
- **计数卡**：采集 / 蒸馏 / AI 创作 / Skill 优化 各 N 次（可选时间范围：近 7 / 30 天 / 全部）。
- **平均耗时**：每类 succeeded 任务的平均时长（`updated_at − created_at`），展示成"蒸馏平均 3 分 20 秒"。
- **对标库规模**：对标博主数、累计真笔记数。
- **最近活动**：最近 N 条任务的时间线（类型 + 时间 + 结果）。

### 模块 2 · 创作与发布（A 层看租户合计，B 层看分账号）
> 计数本身和账号无关，所以 A 层总能展示租户合计；绑了账号的创作再额外计入对应账号的 B 层。
- **AI 创作了 N 篇**（`XhsPublishPackage` 计数，确定的）。
- **已发布 M 篇**（用户反馈，见下方"发布反馈入口"）。
- **创作→发布转化率 M/N**。
- 发布反馈入口（关键交互）：
  - 在每条草稿/作品上放**「我已发布」按钮**，点击记 `published_at` + `status='published'`，并让用户选/确认是哪个"我的账号"。
  - 草稿列表顶部提示：「复制内容去平台发布后，记得回来点『我已发布』，这样才能统计真实效果」。
  - 可"撤销已发布"（手滑兜底）。

### 模块 3 · 涨粉趋势 + 归因（B 层，依赖每日快照）
**归因难点：我们无法证明涨粉是 PubSync 的功劳，所以不假装能。** 用三件事叠加，让相关性自己说话：
1. **涨粉曲线**：按 `account_metric_snapshots` 画粉丝数随时间变化（也可切换"笔记总互动"）。
2. **使用事件打点**：在曲线上叠加竖线标记——哪天蒸馏 / 创作 / 标记发布。用户肉眼看到"开始用之后这段涨得更快"。
3. **活跃期 vs 沉默期对比**：有 PubSync 活动的周 平均日涨粉 vs 没活动的周。
- **必须的免责文案**：「以下为相关性参考，非因果证明；最终效果取决于内容本身。」——诚实比夸大更可信。

### 模块 4 · 产能与省时（B 层，"帮助"的量化）
- **产能**：本月为该账号创作 M 篇、采集/蒸馏支撑了多少准备工作。
- **省时**：`省时 ≈ AI创作篇数 ×(手写一篇估时 − AI起草估时) + 蒸馏次数 × 研究对标估时`。
  - 估时常量入 `config.py` + `admin/config_registry`（后台可调）：`minutes_write_per_post`、`minutes_ai_draft_per_post`、`minutes_research_per_distill`。
  - 展示成"本月约帮你省 6 小时 / 多产出 12 篇"。

### 模块 5 · 相似度趋近（B 层）
- `SkillTrainingRun` 的 `gap_closed%`（floor→ceiling 趋近程度）随时间/优化次数的曲线，叠 floor/ceiling 两条参照线。
- 绑该账号对标的博主；多博主则多条线或可切换。

### 模块 6 · 其他佐证指标
- **爆款率**：互动高于该账号自身均值 X 倍的笔记占比。
- **图文 vs 视频**：哪个模态篇均互动更高（互动已按模态归一，见采集模块）。
- **合规通过率**：创作内容合规扫描的通过比例。

- `published_at: datetime|None` —— "我已发布"反馈时间；`status` 增加取值 `published`。

## 六、每日快照定时任务
- 复用现有"刷新博主"链路（`blogger_distillation` 已有 `user_info`/`follower_count` 拉取，见 `service/collection.py`、`search.py`）给每个 `account_type='mine'` 的账号拉当前指标。
- 调度：用现有任务/定时机制（与 `daily_publish` 同套），每天一次；幂等 upsert 到 `account_metric_snapshots`（同日重跑覆盖）。
- 失败/限流：单账号失败不影响其它账号；记事件，缺一天就缺一个点（曲线容忍断点）。
- 成本：每账号每天一次 `user_info` 调用，量很小。

## 七、后端 API
- `GET /dashboard/overview`（A 层）：各 task_type 计数 + 平均耗时 + 对标库规模 + 最近活动；支持 `range=7d|30d|all`。
- `GET /dashboard/account/{account_id}`（B 层）：创作/发布数与转化率、产能省时、相似度趋近、爆款率/模态/合规；支持 range。
- `GET /dashboard/account/{account_id}/growth`：涨粉时序（snapshots）+ 使用事件打点 + 活跃/沉默对比。
- `POST /xhs/publish-packages/{id}/mark-published` / `unmark-published`：发布反馈（带 my_account_id 校验）。
- 创作接口（`xhs_package_draft`）入参加 `my_account_id`（必选/可选按拍板）。

## 十一、验证
- 后端单测：各 task_type 计数/平均耗时聚合正确；发布反馈改 status/published_at；快照幂等 upsert（同日重跑不重复）；省时公式按常量算对；分账号隔离（A 账号看不到 B 账号数据）。
- 真机：创作时选账号→草稿带账号；点"我已发布"→发布数+1、转化率更新；连续两天快照→涨粉曲线两点 + 事件竖线；切换账号数据隔离。
- 按 CLAUDE.md：ruff + pytest + build → 提交（无 co-author）+ push → `deploy.sh update`（有迁移）→ 看日志确认 0015。

## 十二、不做的事（本轮）
- 不展示金额成本（归后台 admin）。
- 不做"PubSync 内容 vs 自己原创内容"的逐篇互动归因（要把发布反馈匹配到真实笔记，太重）——远期可选。
- 不接第三方数据看板；不导出报表（先站内看）。
- 不宣称因果，只呈现相关性。
