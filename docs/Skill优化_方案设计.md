> 定稿 2026-06-24。**纯业务逻辑**设计,不含实现代码。参考开源库 SkillOpt(`/Users/pinjhu/work/personal/opensource/SkillOpt`,microsoft,MIT,v0.1.0)的真实 API 与训练纪律,结合 PubSync 自身业务。
>

>
> 定位:对标 `商业模式与定位.md` —— Skill 优化是**进阶档卖点/壁垒,不是 MVP**;先把一次性蒸馏的 skill 跑通价值,再上"把 skill 练得更像 + 让用户看到提升"。

---

## 1. 这个模块要干什么

蒸馏给出的是**一次性、靠强模型一把生成**的 skill(博主打法的 8 章 markdown)。它够用,但不保证"贴近博主真实风格"、也不会随反馈变好。

**Skill 优化** = 用**博主自己的真实笔记**当训练信号,像训练神经网络那样(epoch / 验证门控 / 学习率)**只改 skill 这份文本**、不动模型权重,把它练得**更像该博主**;并让用户**直观看到优化前后的提升**,形成正反馈。

一句话:**"把对标 skill 从'像个大概'练到'以假乱真',且每次优化都有据可证。"**

---

## 2. 参考 SkillOpt:借什么、不照搬什么

SkillOpt 的核心主张正是我们要的:**把 skill 文档当成"冻结的目标模型"的可训练状态**,用一个独立的优化器模型把"打了分的 rollout"变成对 skill 的**有界 add/delete/replace 编辑**,且**候选编辑只有在留出验证集上严格变好才被采纳**;部署产物是一份紧凑的 `best_skill.md`,**推理期零额外模型调用**。

**借它的纪律(直接采用):**
| SkillOpt 概念 | 真实位置 | 我们怎么用 |
|---|---|---|
| 六步训练循环 rollout→reflect→aggregate→select→update→evaluate | `skillopt/engine/trainer.py` `ReflACTTrainer.train()` | 整套照搬 |
| 文本梯度 = 对 skill 的有界编辑 | `skillopt/gradient/` | 照搬:优化器模型产出 edits |
| **验证门控**:候选必须在 held-out 验证集严格变好才采纳 | `skillopt/evaluation/gate.py` `evaluate_gate(...)` | 照搬,这是不退化的关键 |
| 学习率(每步最多几处编辑)+ 调度 + 拒绝缓冲 + epoch 慢更新 | `optimizer.learning_rate` 等 | 照搬,保证稳定 |
| MiniMax 后端 | `skillopt/model/minimax_backend.py` `configure_minimax_chat()` | 我们底座就是 MiniMax-M2.7,直接用 |
| env = `dataloader.py` + `rollout.py` + 种子 skill,rollout 返回 `{id, hard(0/1), soft(0-1)}` | `skillopt/envs/searchqa/` | **照葫芦写我们自己的 env** |

**不照搬什么(我们和它的根本差异):**
- SkillOpt 内置 benchmark 是**有对错的任务**(QA 的 EM/F1、代码能否跑通)→ score 客观自带。
- **我们没有"对错",只有"像不像"** —— 目标是"生成的内容贴近博主风格"。所以 **rollout 的 score 要我们自己定义**(见 §4),这是整个方案最核心、也最容易踩坑的地方(PoC 已踩过)。

## 3. 把我们的业务映射到 SkillOpt 的抽象

| SkillOpt 抽象 | 我们的对应 |
|---|---|
| 可训练 skill 文档 | `BloggerSkill.skill_markdown`(蒸馏产出的 8 章打法) |
| 种子 skill(`initial.md`) | 该博主**当前 active 的 skill**(优化的起点) |
| env 数据集 | 该博主的**真实笔记**(`BloggerPost`),按时间留出划分 train/val/test |
| 一条样本 | 一篇真笔记 → 取其**选题**当输入,其**正文/口播稿**当"gold 风格参照" |
| rollout(用 skill 做一次) | 用当前 skill + 该选题,走**我们现有创作链**生成一条(对齐模态) |
| rollout 的 score(hard/soft) | **风格接近度**(StyleDist 为主,客观)+ **归属判别**(hard);详见 §4 |
| 验证门控的 metric | val 集上的 StyleDist(`gate_metric` 用 soft 或 mixed) |
| 部署产物 best_skill.md | 优化后的新 skill 版本(经门控、确实更像) |

## 4. 训练信号(score)——本方案的核心

PoC 的关键教训(问题清单 #10):**MiniMax 评委给了 67-70 分,但客观 StyleDist 只有 1.4(比其它博主的 2.1 还低)——评委被骗了**。结论:**绝不能只用 LLM 评委当训练奖励**。所以 score 以**客观指标为主、评委只作辅助诊断**。

### 4.1 主信号:StyleDist(风格指纹距离,客观、便宜、确定性)
把一段文本压成一个**风格指纹向量**(纯文本统计,不调 LLM):
- 句长分布、段落数/长度、标点频率(感叹号/问号/省略号)、emoji 密度;
- 标题套路(数字开头?疑问句?字数);
- 高频词 / n-gram、口头禅、人称(第一人称密度);
- 列表/分点比例、CTA 句式、话题标签数量。

`StyleDist(a, b)` = 两个指纹向量的距离(越大越远)。**风格接近度 = 生成内容指纹 vs 博主真实笔记指纹的接近程度**。

### 4.2 锚点电池(让分数可解释 + 防自欺)
单看 StyleDist 没有标尺。每次优化先量三根锚:
- **floor**:同赛道**其它**博主真笔记 vs 本博主指纹的距离(随便学学能到的下限);
- **ceiling**:本博主**另一批**真笔记 vs 自己指纹的距离(同一个人不同篇之间的天然距离 = 天花板);
- **whiteboard**:白板 AI(无 skill)生成的距离(完全不学的起点)。

**gap_closed%** =(生成接近度 − floor)/(ceiling − floor) —— "离'以假乱真'还差多少",这是给用户和门控看的核心数。

### 4.3 hard 信号:归属判别(0/1,门控友好)
"把生成内容、本博主真笔记、其它博主真笔记混在一起,判别器(可用指纹最近邻,或一次轻量 LLM 判别)能不能把生成内容**归到本博主名下**" → 能=1,不能=0。对应 SkillOpt 的 `hard`。

### 4.4 LLM 评委(SFS):只作辅助,不进门控
保留一个"像不像"的 LLM 打分,但**只展示、不当训练奖励/门控**(PoC 证明它会被骗)。它和客观指标背离时,正好提示"skill 在讨好评委而非真的变像"。

### 4.5 模态对齐(PoC 的另一坑)
PoC 里博主真笔记是**视频口播逐字稿**,而 skill 生成的是**图文稿**,两种文体字符层面几乎不重叠 → StyleDist 失真。**对策**:rollout 产出的内容**模态必须与 gold 一致**——
- 博主是视频号(`scope_json` 标 talking_video)→ 生成**口播脚本**,gold = 真笔记 transcript(**去时间戳**);
- 博主是图文号 → 生成图文正文,gold = 真笔记正文。

`BloggerSkill.scope_json` 已记录模态,据此决定 rollout 产物形态和 gold 取法。

## 5. 数据与划分(防泄漏、防过拟合)

- **来源**:该博主已采的 `BloggerPost`(active)。**最低样本**:不足阈值(建议 ≥ ~20 篇)不允许优化,提示"先去采集更多笔记"。
- **时间留出划分**(温度计:防 skill"背答案"):按 `published_at` 排序 →
  - 最旧一批 → **train**(给 reflect 当反思素材);
  - 中间 → **val**(门控用,skill 没见过);
  - 最新 → **test**(最终报告,完全没参与训练)。
  对应 SkillOpt 的 `SplitDataLoader` 的 train/val/test,但**按时间切而非随机**,避免"用学过的笔记给自己打高分"。
- 模态混杂的博主:按主模态切,少数派模态的笔记进 train 但不进 val/test 打分(避免口径不一)。

## 6. 产物与版本

现状:`BloggerSkill` 每博主只有一个 active(蒸馏产物),没有版本历史。优化需要**版本化**:
- 每次优化产出一个**新 skill 版本**(seed 的后代),记录:来源 run、训练配置、test 集前后指标、gap_closed、采纳/放弃。
- **门控**:val 没严格变好 → 这次优化**不采纳**(seed 保持 active),如实告诉用户"这次没练出提升"。变好 → 新版本可**采纳为 active**(老版本归档,可回滚)。
- 用户能看到该博主的 **skill 版本时间线** + 每版的效果数,可回滚。

## 7. 用户流程

**入口**:侧栏「评估与提升 · Skill 优化」(已占位)。也可从博主资产/蒸馏页某个 skill 上点「优化这个 Skill」。

**前置条件**(任一不满足就地提示):
1. 该博主有 active skill(没有 → 先去蒸馏);
2. 笔记数 ≥ 阈值(不足 → 先去采集);
3. 模态口径可对齐(视频号确认走口播脚本)。

**流程**:
```
选一个对标博主(已有 skill + 够样本)
  → 一键「优化 Skill」(异步任务,接 LiveProgress)
  → 后台:划分数据 → 量锚点 → 跑 SkillOpt 循环(rollout/reflect/门控)→ test 报告
  → 回到前端:前后对比卡(见 §8)
  → 用户「采纳新版」或「保留原版」(门控没过则只给"未见提升",不强推)
```

**进度展示**:复用统一进度时间线(划分数据 / 跑基线 / epoch k 候选评分 / 门控结果 / 出报告),通俗文案。

## 8. 效果可视化 & 正反馈(模块价值的落点)

**优化前后对比卡**(采纳决策的依据,也是"让用户看到帮助"的直接证据):
- 三根指标并排:**gap_closed%**(核心)、可辨别度(越低越像)、StyleDist;每项 `优化前 → 优化后(+Δ)`。
- **样例对照**:同一选题,种子 skill 生成 vs 优化后生成 vs 博主真笔记,三栏并排,让用户**肉眼看出更像了**。
- 门控结论:✅ 采纳(val 严格变好)/ ⚠️ 未见提升(本次不采纳,附可能原因:样本少/模态杂)。

**正反馈回路(分期接入)**:
1. **离线**(本模块):优化前后对比,即时正反馈。
2. **在线**(接 `效果看板`):用优化后 skill 创作 → 发布 → 真实涨粉/互动回流 → 进对标差距收敛曲线,证明"练过的 skill 带来真实增长"。
3. **自动重训触发**(后期):对标博主上新爆款 / 我的账号数据更新到一定量 → 提示"可重训",形成长期循环。

---

## 9. 何时(重新)优化

- **底座变更**:skill 是绑定底座调出来的;换大模型 → skill 优化有偏差,需重训(`Skill训练_SkillOpt方案设计.md` 已论证)。
- **对标演化**:对标博主风格/选题明显变了(新采集触发)。
- **数据增量**:新采了一批高赞笔记,训练信号更足。
- 默认**不自动跑**(费钱),由用户手动触发或系统给"建议重训"提示。

## 10. 集成方式:子进程跑 SkillOpt + 自定义 env(推荐)

SkillOpt **文件系统强耦合**(必须写 out_root/split_dir)、**全局模型配置态**(`configure_minimax_chat` 是进程级)。因此:

- **方式 A(推荐):子进程 + 自定义 env 包**。我们写一个 `envs/pubsync_style/`(dataloader = 从我们 DB 导出该博主笔记的时间留出划分;rollout = 调我们现有创作链生成 + StyleDist/归属打分;种子 = 当前 active skill),worker 起一个子进程跑 `ReflACTTrainer.train()`,训练目录用临时区,跑完读回 `best_skill.md` + `history.json` 落库。**隔离干净、不污染主进程、贴合库的设计**。
- **方式 B:自研精简循环**。完全照搬六步纪律但自己写(不依赖 v0.1.0 这个新库当运行时)。更可控、无外部依赖风险,但要自己维护循环正确性。

**建议**:**M1 用方式 A 跑通真实提升信号**(库成熟、省自研成本);若验证有效且要长期产品化,**M2 视库稳定性决定是否转方式 B**(把循环自研、藏在同一接口后)。两种方式对前端/DB/任务层完全一致,可后端内部替换。

> 数据出 DB → 喂子进程,**绝不把生产密钥/隐私写进训练目录**;训练在容器内、密钥走容器环境变量(沿用现有约定)。

### 10.1 "子进程"到底是什么(不是单独部署)
SkillOpt 是 **pip 库**,装进**现有 worker 容器**即可——不开新服务/容器/端口。worker 接到训练任务时在**同机 fork 一个子进程**跑 `ReflACTTrainer.train()`,跑完读回结果。用子进程而非 in-process 只为:隔离库的进程级全局配置(`configure_minimax_chat`)、能干净设超时/取消。**部署动作 = 给 worker 镜像加依赖,无新增基建。**

### 10.2 进度/日志到前端(复用 LiveProgress)
rollout 的 env adapter **由我们编写**,所以每步(划分数据 / 跑基线 / epoch k 候选评分 / 门控通过|拒绝 / 出报告)由 adapter **主动写 `operation_task_events`**——即 LiveProgress 时间线现用的表。前端那条进度时间线直接显示训练过程,**零新增前端基建**;不解析库的 stdout(脏),而是发我们掌控的结构化事件。`history.json` 作为兜底回读。

### 10.3 轻量服务器可行性(腾讯云轻量级)
SkillOpt **不在本地跑模型**,是"调 MiniMax API + 本地 numpy 文本统计"的编排层。其**核心依赖极轻**:`openai / httpx / numpy / pyyaml / openpyxl / azure-*`(几十 MB,**无 torch / CUDA / vLLM / Ray / transformers**;重依赖全是 optional,仅"本地 Qwen""ALFWorld"才需要,我们不装)。因此训练是 **IO 密集(等 API)非算力密集**,轻量机可跑。约束:
- **并发调小**:rollout `workers` 默认 64 → 改 **4–8**(进 admin 可调),慢但稳,不冲内存/网络/API 限频。
- **串行**:一次只跑一个训练任务(RQ 队列天然串行),避免抢 backend/采集的 CPU。
- **内存**:numpy+文本,几百 MB 级。**2C2G 偏紧、2C4G 舒服**;训练时段其它任务可能略慢。
- 临时训练目录几 MB,跑完即清。
- **方式 B(自研循环)** 同样只需 numpy+API,footprint 更小、无外部依赖——若想把轻量机吃干榨净,B 也是好选择。

## 11. 成本

- 一次优化 ≈ 几十次生成(rollout)+ 几十次优化器编辑(reflect)+ 门控评分,**StyleDist/锚点是本地计算(免费)**。PoC 实测 ~$0.03/次(MiniMax M2.7)。
- 不抓新数据(用已采笔记)→ **零 TikHub 成本**。
- 成本闸:epoch 数、batch、edit_budget、val 样本数(都进 admin 可调);默认小循环,够用即停。
- 定位进阶档:成本可忽略,卖点是"可量化的提升 + 正反馈",不是算力。

## 12. 风险与对策(含 PoC 教训)

| 风险 | 对策 |
|---|---|
| **只信 LLM 评委 → 被讨好型 skill 骗**(PoC 实证) | 门控只用客观 StyleDist/归属;评委仅展示 |
| **模态错配**(视频稿 vs 图文)致指标失真(PoC 实证) | rollout 产物与 gold 严格同模态;视频去时间戳 |
| 样本太少 → 过拟合 / 训不动 | 最低样本阈值;时间留出;val 门控挡退化 |
| 优化无提升却强推给用户 | 门控没过就如实显示"未见提升",seed 保持 active |
| 库 v0.1.0 太新、当运行时依赖有风险 | 子进程隔离;保留转自研循环的退路(§10) |
| 用户不懂指标 | 主打"样例三栏肉眼对比 + gap_closed 一个数",指标细节藏在展开里 |

## 13. 分期 & 不做的事

**分期**:
- **M1 · 干净 PoC**:在我们 env 里用 StyleDist 当奖励、模态对齐,挑一个图文博主 + 一个视频博主各跑一次,确认能拿到"训练有效"信号(PoC 上次被数据问题污染,没拿到干净信号)。
- **M2 · 产品化**:skill 版本化 + 优化任务 + 前后对比卡 + 采纳/回滚 + 「Skill 优化」页落地。
- **M3 · 在线正反馈**:接效果看板差距收敛曲线;自动重训建议。

**本模块不做**:
- 不动模型权重(只训 skill 文本);
- 不在优化里抓新数据(只用已采笔记);
- 不做自动无人值守重训(先手动触发);
- 不把 LLM 评委当训练奖励。
