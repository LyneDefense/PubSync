# 蒸馏车道化方案（一个 Skill：共享内核 + 内容层按模态分车道）

## 0. 一句话
把「一个巨型 prompt 一次吐 18 字段」的蒸馏，改成 **1 次内核蒸馏（人）+ 每条"样本够"的模态车道各 1 次内容蒸馏（表达）**。
模态由已上线的 `content_subtype`（图文/口播/非口播 + confidence）做路由。顺带治好"注意力摊薄"的老毛病。

## 1. 为什么这么切
- **认知层 + 策略层 + 人设/价值立场 = 「人」**：跨模态一致（一个人的信念/选题逻辑/人设不随格式变）→ pool 全部笔记，样本永远够。
- **内容层 = 「表达」**：图文是书面语+正文结构、口播是口语+口播脚本、非口播是画面/节奏——三种写法根本不同 → 按车道各自深挖。
- 下游（创作按模态发、Skill优化按模态 rollout）本就分模态，蒸馏这一段过去却把车道并了道，是断点。此方案把它接上。

## 2. 路由（核心）
一次蒸馏的选中笔记池 `posts`：
```
内核池   = 全部 posts（含 unknown 视频——判不了模态，但标题/互动/评论仍是跨模态信号）
车道池   = posts 按 content_subtype 分组:
          image_text / talking_video / visual_video
          （unknown 不成车道）
```
- 每条车道 `len ≥ settings.distill_min_samples_per_subtype`（默认 5）才生成；不足 → 跳过 + 记「该车道样本不足(仅N篇)，未出写法」。
- **选材保持模态无关**：用户按相关度选笔记，分流交给蒸馏；结果回报各车道覆盖数。

## 3. 管线（方案 A：内核 + 每车道各一次）
```
analyze_posts(全部)            → 内核 stats
run_synthesis(内核 guide)      → 内核结果(认知/策略/人设/…)      [1 次 +可修订]
for lane in present_lanes:
    analyze_posts(lane_posts)  → 该车道 stats
    run_synthesis(lane guide)  → 该车道内容层                    [每车道 1 次 +可修订]
合并 → 一个 distillation 结果 → 一个 skill + 一份报告
```
- 复用现有 `run_synthesis`（生成→校验→修订循环）：内核一个 guide，车道一个参数化 guide。
- 典型号 1-2 条车道（阿甜基本一条）→ 2-3 次调用；GLM 5.2 快，扛得住。

## 4. Prompt 拆分
- **内核 prompt**（`build_core_prompt`）：`one_glance / persona / audience / cognitive_layer / strategy_layer / value_stance / growth_trend / comment_insights / do_not_do / sample_topics / core_conclusion`，模式 B 另加 `self_diagnosis`。喂全账号 stats。**不含内容层**。
- **车道 prompt**（`build_lane_prompt(lane, lane_stats)`）：该模态的 `内容层`——
  - 图文：`title_formulas / opening_templates / body_structures / cover_text_rules / language_dna(书面) / emotional_rhythm / cta_strategy / hashtag_strategy / top_post_breakdowns`
  - 口播：把 `body_structures` 换 `video_script_structures`（开场钩子/信息密度/讲述节奏/结尾），`language_dna(口语)`
  - 非口播：**诚实边界**——文本能给的只有 `title_formulas / cover_text_rules / hashtag_strategy / 节奏(从元数据)`；画面/卡点 craft 标「需人工或视觉模型」，不硬编
  - 每条车道只喂**该车道的 stats + 该车道的爆款样本**（top_post_breakdowns 拆该车道的高赞）。

## 5. 数据结构变化
蒸馏结果 JSON：内核字段保持扁平；内容层从单个 `content_layer` 改为按车道分组：
```jsonc
{
  "one_glance": "...", "persona": {...}, "cognitive_layer": {...}, "strategy_layer": {...},
  "growth_trend": "...", "comment_insights": [...], "do_not_do": [...],
  "self_diagnosis": {...}, "core_conclusion": "...", "sample_topics": [...],
  "content_lanes": {
    "image_text":   { "sample_count": 12, "title_formulas": [...], "body_structures": [...], "language_dna": [...], ... },
    "talking_video":{ "sample_count": 8,  "title_formulas": [...], "video_script_structures": [...], ... }
    // 只列 present 车道
  },
  "lane_coverage": { "image_text": 12, "talking_video": 8, "visual_video": 2, "unknown": 3,
                     "skipped": ["visual_video(仅2篇,<5)"] }
}
```
- 老 run 仍是旧形态（扁平 `content_layer`）——**报告/技能渲染对两种形态都兜底**（有 `content_lanes` 用新的，否则回退旧的），不做数据迁移。

## 6. 质量分（按车道）
- 内核 rubric：认知层≥4、策略层≥2、（模式B）自我诊断≥3。
- 每条 present 车道各自 rubric：标题公式≥3、该模态结构非空、爆款拆解≥min(3,该车道hot)、来源标注、空泛检测。
- **缺席车道不扣分**（样本不足 ≠ 质量差）。总分 = 内核分 × 各车道分的合理加权（或分别展示，不强求一个总分）。

## 7. 产物
- **Skill md**：`使用说明 → 一、认知层 → 二、策略层 → 三、内容层(按车道分节:3A图文写法/3B口播写法/3C非口播写法,只列present) → 创作禁区 → 对比示例 → 选题灵感 → 局限(注明各车道样本数/缺席)`。人设/价值立场在内核(谁)；语言DNA/写法在车道(各格式怎么写)。
- **HTML 报告**：「内容公式速查」模块改为**按车道分 tab**；其余模块不变。
- `scope_json` 天然 = present 车道集合（已有机制，对齐）。

## 8. 全局缝合
- **创作**（后续小步）：创作 agent 按平台×内容类型，读 `content_lanes[对应车道]` 而非整坨内容层；发口播就取口播车道。
- **Skill优化**（未来）：rollout 已按 talking_video/image_text 分车道；skill 分车道后可精准优化某条车道写法。本方案不改 skill-opt，只让它未来能对齐。

## 9. 关键文件（实现时）
- `blogger_distillation/service/distill_engine.py`：拆 `build_distill_prompt` → `build_core_prompt` + `build_lane_prompt`；`distill_with_llm` 改为「内核 + 循环车道」编排；normalize/quality 按新结构。
- `blogger_distillation/service/distillation.py`：路由（内核池/车道池/≥5 门控）、多次 run_synthesis、合并结果、`lane_coverage` 事件。
- `blogger_distillation/analysis.py`：`analyze_posts` 已是纯函数按列表算，直接对子集调用即可（可能补「该车道爆款」口径）。
- `blogger_distillation/artifacts*`：skill md 分车道节 + HTML 报告分车道 tab + 对旧形态兜底。
- `schemas/blogger.py`：蒸馏结果读模型加 `content_lanes` / `lane_coverage`（前端渲染用）。
- 前端 `AppraisalCard`/报告渲染：内容公式速查按车道 tab；历史旧报告兼容。
- config：无新增（复用 `distill_min_samples_per_subtype`）。

## 10. 验证
- 单测：路由(unknown 只进内核、<5 车道被跳过)、内核/车道 prompt 各自字段、按车道质量分、旧形态渲染兜底。
- 真机：挑一个**含图文+口播两种**的号 → 出的 skill 有两条车道节、语言DNA书面/口语分开；挑阿甜(基本单车道)→ 只出一条车道 + 明确标其余车道样本不足。
- 按 CLAUDE.md：ruff + pytest + build → 提交 → `deploy.sh update`(如动 schema 无迁移则 update-frontend/backend) → 验证。

## 11. 不做（本轮范围）
- 不做 grounding（带出处候选表 + 引用校验）——与车道化正交，作为**下一层**叠加到内核/车道 prompt（之前讨论的 Phase 1，可车道化之后再上）。
- 不改创作/Skill优化的内部（只预留读车道的口子）。
- 不做选材的车道锁定（保持模态无关）。
- 不迁移老 run 数据（渲染兜底即可）。
