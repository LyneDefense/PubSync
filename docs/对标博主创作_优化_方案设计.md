# 对标博主创作 · 优化方案设计

> 状态:设计定稿待确认 · 2026-07-07
> 范围:小红书「对标博主创作」(发布包创作)模块。抖音同构复用;公众号仍占位。

## 一、背景(为什么改)

这个模块**一直没动**,但它周围的世界全变了:

- **删掉的功能**:手挑笔记蒸馏 / 命名快照、Skill 优化(训练)、效果看板、智能推荐 + 单博主评分、发现会话。→ 创作画像现在**一个博主只有一份**(去多画像,建档自动蒸馏),不再有"多 Skill / 快照"。
- **新增的数据**:每个对标博主背后现在有一整套**档案** —— 全量笔记池、数据/成长轨迹、运营习惯、**受众需求(读者最常问)**、**合规红线**、**分车道创作画像**(图文 / 口播 / 非口播)、真爆文池。而且**「我的账号」也有了档案 + 体检**。

**问题**:创作模块是"旧世界"(手挑笔记蒸多个薄 Skill)的产物,**几乎没用上这些新数据**。尤其选题环节纯靠方法论空想,和"我自己的读者要什么"完全脱节。

## 二、现状(对着代码)

**前端**:`SocialPackagesView.vue` 六步向导 —— ① 选博主 → ② 选 Skill → ③ 选题 → ④ 生成正文/脚本 → ⑤ 素材 → ⑥ 发布包。

**后端**:`app/xhs_creation/service.py` + `app/xhs_creation/agent/`。

- **选题** `generate_xhs_topic_ideas`(service.py:186):只喂 `skill.skill_markdown[:10000]` + 用户手打的 `target_audience/content_goal/keywords`。**没碰任何档案数据**(受众/爆文/分车道全没用)。
- **正文** `generate_package_content`(service.py:262):经 `_load_report` 吃了蒸馏报告 `stats`(含 hot_posts)+ 结构化 `distillation`(含 content_lanes / 分车道 + compliance_watchouts)。相对扎实,但**用得不满**(few-shot 爆文、合规红线没显式喂)。
- **内容类型**(4 种,`schemas/xhs.py` + `agent/content_types.py`):
  - `text_note` 纯文字;
  - `image_note` 图文:输出 `suitable_image_count` + `image_plan`(每张 `slot/purpose/caption/prompt(英文生图)`),**且 service.py:68-78 会真调 `generate_image` 把图生成出来**;
  - `spoken_script` 口播:`script.segments`(`voiceover` 口播原话 / `subtitle` 字幕)+ `shooting_notes`;
  - `video_script` 视频:`script.segments`(`scene` 分镜 / `voiceover` / `subtitle`)+ `shooting_notes`。
- **② 选 Skill 步骤已基本废**:去多画像后一个博主只有一份 active 创作画像,这步永远单选;且措辞"Skill / 风格资产"与档案「创作画像」不一致。

**关键结论**:口播/视频**早就输出**"口播逐字稿 + 字幕 + 分镜 + 拍摄建议";图文**早就规划**图片方案(含英文 AI prompt),**只是多做了一步真生图**。所以"输出建议而非直接生成"改动很小。

## 三、目标流程(瘦身后)

```
选对标博主(自动带其唯一创作画像) + 选「我的账号」
        ↓
选题(对标博主选题方法 × 我的账号受众需求 [× 意图])
        ↓
选内容类型(对齐创作画像车道)
        ↓
生成(正文/口播脚本;图文=图片方案不生图;爆文 few-shot + 合规前置)
        ↓
发布包
```

## 四、逐项改造

### A · 选题重构(核心,吃新数据)

**逻辑**:选题 = **对标博主的选题方法** × **我的账号的受众需求** [× 意图]。
- 填了意图 → 三者结合(方法向对标学、写什么由"我的受众 + 意图"定);
- 没填意图 → "我的受众需求 × 对标方法"。

> 纠偏:对标博主的受众需求是**对标博主读者**关心的,不能当"我要写什么"。"写什么"的需求信号来自**我自己**的读者。

**数据来源**:
- 对标博主选题方法:`skill.skill_markdown` 的选题思路 + `_load_report` 的 `distillation` 选题层 + `stats.hot_posts`(真爆文标题/角度)。
- 我的账号受众需求:我的账号(`my_blogger_id`)档案的 audience —— 存在 `BloggerProfile.attribution_json`,`app/blogger_dossier/audience.parse_audience` 解析。

**后端**:`generate_xhs_topic_ideas` 入参加 `my_blogger_id`;组装 prompt = 对标方法 + 爆文角度 + 我的受众需求 + 意图。Schema `XhsTopicIdeaRequest`(schemas/xhs.py)加 `my_blogger_id`。

**前端 step 选题**:显示"**你的**读者最关心:X / Y / Z"(取自我的账号档案受众)做**一键种子**;意图输入框保留(可选)。

**降级**(见决策 1):我的账号没绑 / 没受众需求 → 退回"对标方法 + 意图",顶部提示"绑个有档案的我的账号,选题更懂你读者"。

### B · 流程瘦身(对齐去多画像)

- **合并 ①选博主 + ②选 Skill → 「选对标博主」**(自动用其唯一 active 创作画像;`xhsCreatorSkillOptions` 只剩一条时直接选中、不再单列一步)。
- 措辞统一:"Skill / 风格资产" → **「创作画像」**。
- **前置「为哪个我的账号创作」选择**(A 要用我的受众;也是发布回填账号)。`xhsPackageForm.my_account_id` 从"保存时才绑"提前到选题前选。
- 空态引导:对标博主没建档/没蒸画像 → 一键跳「博主档案」建档。

### C · 内容类型对齐车道

`content_type` ↔ 创作画像车道(`distillation.content_lanes`,键 `IMAGE_TEXT / TALKING_VIDEO / VISUAL_VIDEO`):
- 图文 → 图文车道;口播脚本 → 口播车道;视频脚本 → 非口播视频车道。
- 生成时喂**对应车道**的写法(而非整份混着);该车道 `lane_coverage=0`(博主没这类内容)→ 前端提示"该博主没有此类内容,风格参考有限"。

### D · 输出形态改造(产物本身)

**图文(image_note):不再自动生成图片。**
- 删 `service.py:61-78` 的 `generate_image` 循环 + "配图"进度事件 + `image_urls`;`image_note` 的产物 = **图片方案**。
- `image_plan` 每张已含 `purpose`(内容)/`caption`(图上文案)/`prompt`(英文 AI 生图 prompt);**补一个 `format` / 版式字段**(如"竖版 3:4 · 封面 / 步骤图 / 对比图")。`content_types.py` 的 `image_note.schema_fragment` + `normalize.py` 加该字段。
- 前端「素材」步:从"展示生成的图" → 展示**图片方案卡**(第 N 张:版式 · 内容 · 图上文案 · 英文 prompt + 复制按钮)。

**口播 / 视频:保留现有输出**(口播逐字稿 + 字幕 + 分镜 + 拍摄建议),前端把 `script.segments` / `shooting_notes` 展示清楚、可复制。

### E · 爆文锚定正文 + 合规前置

- 正文生成(`build_creation_prompt` / CreationContext)显式喂 **1-3 篇对标博主真爆文的开头 / 结构 / 选题思路**(来自 `stats.hot_posts`,**学思路不抄词**)当 few-shot。
- 显式喂档案 **合规红线**(`distillation.compliance_watchouts`)让它**开局就避**,而非只在 `scan_creation` 事后标红。draft 已吃报告,是"用得更足"。

### F · 为我所用(部分下沉,深化留后)

- 已下沉进 A:选题用我的受众。
- **留后**:正文也套我的账号定位 / 人设(用对标方法 + 我的受众 + 我的画像重写),工作量大,下一轮。

## 五、关键决策(附建议结论)

1. **我的账号没受众需求时** → **降级不挡路**:退回"对标方法 + 意图",提示绑有档案的账号更准。(不硬性拦着先建档)
2. **自动生图** → **直接删**,不留后台开关。省图像模型成本、符合"给建议不代生成"。
3. **分步** → **P1: B + D**(结构性、见效快、低风险)→ **P2: A + C**(选题重构 + 车道对齐,依赖我的账号受众)→ **P3: E**(附正文侧)。F 深化再下一轮。

## 六、接口 / Schema 变更清单

- `schemas/xhs.py` `XhsTopicIdeaRequest` + `my_blogger_id: int | None`。
- `agent/content_types.py` `image_note.schema_fragment` 加 `format` 字段;`required_fields` 视情况。
- `service.py` 去 `generate_image` 循环(image_note 产物只留 `image_plan`;`image_urls_json` 恒空或移除)。
- 前端 `xhsPackageForm` 把 `my_account_id` 提前;`api/index.ts` 选题请求带 `my_blogger_id`;`SocialPackagesView` 步骤合并 + 素材步改图片方案 + 选题步显示我的受众种子。
- **无数据库迁移**(image_urls 列若还在,保留兜底,不删表/列;走"停止写入"即可)。

## 七、验证

- **后端单测**:选题组装(有/无意图 × 有/无我的受众,四组合走对分支);image_note 去生图后产物含 `image_plan` 且不含 image_urls;`format` 字段规范化;车道映射。
- **真机**:选对标博主 + 我的账号 → 选题贴我的受众(种子来自我的档案)→ 图文出**图片方案不出图** → 口播出脚本 + 拍摄建议;没绑我的账号时降级不报错。
- CLAUDE.md 纪律:ruff + pytest + build → 提交(无 co-author)+ push → 部署(P1/P2 纯前后端无迁移,`update` 全量或分 `update-backend`/`update-frontend`)→ 看日志。

## 八、不做 / 留后

- ⑥ 深化(正文套我的人设 / 定位)。
- 公众号创作(枚举占位,不接线)。
- 视觉模型不接;图不代生成。
