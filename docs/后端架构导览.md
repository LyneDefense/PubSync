# PubSync 后端架构导览

> 面向 code review 的结构地图。路径均相对仓库根(`backend/…`)。
> 最后更新:随 `0025_video_profile` 迁移一代。

## 定位

多租户 **FastAPI** 应用,两条产品主线跑在同一套骨架上:

1. **小红书 / 抖音** —— 博主蒸馏 + AI 创作(主线)
2. **公众号** —— 每日早报流水线

重活(采集 / 蒸馏 / 创作 / 早报)全部走 **RQ + Redis** 异步任务,由独立 **worker** 进程执行。规模:约 2.3 万行 Python / 218 文件 / 39 个测试。

## 技术栈 / 进程

- Web + Worker **两个进程,共享同一份代码**。
- FastAPI(HTTP)+ SQLAlchemy(ORM)+ Alembic(迁移)+ Postgres + Redis/RQ(任务队列)。
- LLM 统一走智谱 GLM(含视觉、ASR),经 `ai_service` 网关。
- 数据源:TikHub(小红书 / 抖音)。

| 进程 | 入口 | 职责 |
| --- | --- | --- |
| Web | `backend/app/main.py` | HTTP API、鉴权、入队、取数 |
| Worker | `backend/app/worker.py` | 消费 `pubsync` 队列,执行后台任务 |

## 分层地图(自上而下)

### ① HTTP 层 · `app/api/`(23 文件)
只做入参校验 / 鉴权 / 入队 / 取数,**不放业务**。共享依赖(当前用户 / 租户 / DB session)在 `api/deps.py`。

- 博主(主对象):`api/bloggers/` → `crud` / `collection`(采集)/ `distillation`(蒸馏)/ `dossier`(建档)/ `recommend`(找对标)
- 创作:`api/xhs.py`(发布包草稿 / 保存 / 选题)
- 体检:`api/account_audit.py`
- 公众号:`news` / `articles` / `workspace`
- 后台:`admin` / `admin_tasks` / `admin_costs` / `admin_config`
- 进度轮询:`api/tasks.py`

### ② 任务编排 · `app/services/`
- ⭐ `services/task_service/runners.py` —— **所有后台任务入口**(采集 / 建档 / 蒸馏 / 创作 / 体检 / 诊断 / 早报)。读它即知「有哪些任务、怎么触发」。
- `task_service/core.py`(任务生命周期 + 事件记录)、`maintenance.py`
- ⭐ `services/ai_service.py` —— **LLM 网关**(608 行;GLM / OpenAI / Anthropic / MiniMax + 视觉)。全站 LLM 都过这。
- `wechat_service`(公众号发布)、`tenant_service` / `auth_service`

### ③ Agent 内核 · `app/synthesis/`
「生成 → 自检传感器 → 修订」循环,被创作 / 体检 / 蒸馏复用。
`agent` + `loop`(装配 + 主循环)· `sensors`(自检 / 阻断)· `budget`(token 预算)· `guide` / `trace` / `progress`。

### ④ 域:小红书 / 抖音(核心)
- ⭐ `app/blogger_distillation/`(8098 行,最重)
  - `tikhub_client/` —— 外部数据源(xhs / douyin API + `parsers/` 解析原始 JSON)
  - `service/` —— 采集 + 蒸馏流水线:
    - ⭐ `collection.py`(633 行:选材 / 增量 / 下架对账)
    - `note_pipeline.py`(单篇分派)· `extract/`(post / video / subtitle 规范化)
    - `asr_step`(转写)· `vision_step`(图文视觉)· `motion_step`(视频拍法:分镜 + 代表帧 VLM)· `tagging`
    - ⭐ `distillation.py`(蒸馏编排:路由 + 内核 + 分车道 + 合并)· ⭐ `distill_engine.py`(蒸馏 prompt 内核 + 质量分)
  - 域逻辑:`modality`(模态 / 口播浓度 / `video_profile`)· `evidence`(证据装配)· `analysis`(互动 / 爆文)· `vision` / `asr` / `video_frames`(底层)· `artifacts`(产 skill md + HTML 报告)· `search`(536 行)· `post_content` / `quality` / `privacy`
- `app/blogger_dossier/` —— 建档:`pool` / `stats` / `trajectory` / `audience` / `habits` / `compliance`
- `app/xhs_creation/` —— 对标创作:`service` + `agent/`(平台 × 内容类型 prompt)+ `my_baseline`(拍法基线)+ `normalize`
- `app/appraisal/` —— 对标诊断报告:`intent` / `hard` / `judge` / `gap` / `service`
- `app/account_audit/` —— 我的账号体检(agent 式)
- `app/compliance/` —— 限流词 / 合规:`registry` / `matcher` / `vertical` / `grade` + `packs/`(分赛道词库)
- `app/benchmark_discovery/` —— 找对标(`engine.py`,已瘦身)

### ⑤ 域:公众号早报(独立流水线)
由 `app/pipeline/orchestrator.py` 串起:
`news_fetching`(抓)→ `news_processing`(清洗)→ `news_deduplication`(去重)→ `article_selection`(选)→ `article_composition`(成文)→ `article_layout`(排版)。
`app/tools/` 是这条线用的旧工具封装(llm / image / article / wechat / storage / news_fetch)。

### ⑥ 数据 / 契约 / 横切
- `app/models/` —— ORM 表(23 张,按域分文件)
- `app/schemas/` —— Pydantic I/O(按域分文件)
- `app/admin/` —— 运行时配置(`runtime_config` / `config_registry` / `secret_box`)
- `app/cost/` —— LLM / API 记账(`pricing` / `context`)

## 后台任务全景(`services/task_service/runners.py`)

| 任务函数 | 触发 | 干什么 |
| --- | --- | --- |
| `run_blogger_collection_task` | 采集博主 | 拉列表 → 增量 diff → 详情/评论/ASR/视觉/拍法 → 落库 |
| `run_blogger_url_collection_task` | 粘链接定向采集 | 单/多条笔记链接精确补采 |
| `run_blogger_dossier_task` | 建档 | 全量池信号 + 自动蒸馏 |
| `run_blogger_distillation_task` | 蒸馏 | 选材 → 内核 + 分车道 → skill + 报告 |
| `run_blogger_redistill_task` | 更新画像 | 用现有详情池重蒸(便宜) |
| `run_blogger_pool_sync_task` | 同步池 | 增量刷新笔记池 |
| `run_xhs_package_draft_task` | AI 创作 | 生成发布包草稿(创作 agent 循环) |
| `run_account_audit_task` | 我的账号体检 | 体检报告 |
| `run_appraisal_task` | 对标诊断 | 诊断报告(benchmark / self) |
| `run_news_fetch_task` / `run_article_generation_task` / `run_daily_publish_task` | 公众号早报 | 抓取 / 成文 / 发布 |

## 一次「蒸馏」怎么流(端到端 trace 样例)

```
蒸馏接口 api/bloggers/distillation.py
  → 入队(queue.py)
  → worker 跑 run_blogger_distillation_task(services/task_service/runners.py)
  → blogger_distillation/service/distillation.py 编排
      → 取笔记池 + evidence.py 装证据
      → distill_engine.py 拼 prompt(core / lane)
      → synthesis/ agent 循环(过 ai_service.py 调 GLM)
      → artifacts.py 出 skill markdown + HTML 报告
      → 存 BloggerSkill + 记 events
  → 前端轮询 api/tasks.py 拿进度
```

创作、体检、诊断是同构的:`api → runners → 域 service → synthesis agent → ai_service → 产物 + events`。

## 建议 review 顺序

1. **骨架**:`config.py` → `main.py` → `services/task_service/runners.py`(先建立"任务全景")
2. **网关**:`ai_service.py` + `synthesis/`(所有 LLM 出口 + agent 循环)
3. **主链路**:`collection.py`(采集)→ `distillation.py` + `distill_engine.py`(蒸馏)→ `xhs_creation/`(创作)
4. **域逻辑**:`evidence` / `analysis` / `modality` / `compliance`
5. **公众号**:`pipeline/orchestrator.py` 一条线读下去

**重点盯的大 / 复杂文件**:`collection.py`(633)、`ai_service.py`(608)、`search.py`(536)、`artifacts.py`(487)、`asr.py`(457)、`distill_engine.py`(436)、`tikhub_client/parsers/`。
