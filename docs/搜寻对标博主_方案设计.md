# 搜寻对标博主 — 方案设计

> 定稿 2026-06-24。范围**只覆盖"找到并确认一个对标博主"**：关键词搜索 / 智能推荐 / 单博主评分三个入口 + 一个共享评估内核。采集、蒸馏、创作、诊断不在本方案内——确认对标博主后才进入采集流程（见 `业务全流程设计_含Skill训练.md`）。
>
> 决策链（来自一次产品讨论的连续修正）：
> 1. "匹配度/相似度 vs 我的账号"**不属于搜索阶段**，它是后面 ⑤ 诊断的评判标准；
> 2. 推荐要解决的是"用户不知道对标谁"——填简单信息，平台帮找并给依据；
> 3. 输入 = **意图（永远必填）〔＋ 可选某个"我的账号"作为增强依据〕**，账号在则更准且解锁"可迁移度"，账号不在则纯靠意图。

---

## 0. 三个入口的定位 + 一个共享内核

三个入口对应三种用户状态，但**最终都汇到同一个"对标评估内核"**，再统一走"确认 → 入库为对标博主 → 触发采集"。

| 入口 | 用户状态 | 一句话定位 | 现状 |
|---|---|---|---|
| **A. 关键词搜索** | "我大概知道搜什么词" | 快速定位 + 按需深评 | ✅ 已有搜索，**缺评估** |
| **B. 智能推荐** | "我不知道该对标谁" | 填意图 → 平台帮我找一批 + 说明为什么 | ❌ 全新 |
| **C. 单博主评分** | "我盯上某个号了，靠不靠谱？" | 给一个号 → 出"适不适合当你的对标 + 依据" | ❌ 全新 |

**共享内核 = 对标评估器** `evaluate(candidate, context)`：输入一个候选 + 用户上下文，输出四项指标 + 综合分 + 判断依据。
- B = 对候选池**批量**跑内核；
- C = 对**单个**候选跑内核；
- A = 搜出来后**按需**对某个候选跑内核。

---

## 1. 用户上下文 `BenchmarkContext`（三入口共用参照系）

构建一次，喂给内核。

```
BenchmarkContext {
  intent {                      // 必填，永远要填（即使绑了账号）
    niche: str                  // 必填：领域/赛道（自由文本，如"保险经纪""母婴""AI工具"）
    audience?: str              // 给谁看（宝妈/职场新人/本地客户…）
    goal?: enum                 // 涨粉立人设 / 私域获客 / 带货
    content_form?: enum         // image_text / video / any
  }
  my_account_id?: int           // 可选；绑了"我的账号"则用它增强（多账号必须选一个）
}
```

- 有 `my_account_id` → 后端从该账号**已采笔记**推断真实赛道/风格/受众画像（复用 `account_audit.build_account_content()` 思路），作为额外信号，并**解锁"可迁移度"指标**。
- 多个"我的账号" → 前端强制选一个。
- 无账号 → 跳过账号增强，纯靠 intent。

**分工对照**：
- **① 推荐/评分**：依据 = 意图（必填）+ 可选某个"我的账号"；账号在则更准、多给可迁移度；账号不在则纯意图。
- **⑤ 诊断**（已实现，不在本方案）：选定对标后，"我的账号 vs 对标"逐维度深比、出差距与动作。

## 2. 数据与成本策略（小红书详情 $0.01/条，无折扣 → 必须卡死）

**铁律：评估阶段绝不批量抓详情。**

| 数据层 | 内容 | 成本 | 用在 |
|---|---|---|---|
| 搜索返回 | 昵称/粉丝/简介/头像 | 含在搜索调用 | 全部入口 |
| 列表级 | 近期 N 篇标题 + 点赞/互动 | 便宜（列表接口） | 评估打分 |
| 详情级 | 单篇正文/字幕/全互动 | **贵** | 仅"深看" / 确认后正式采集 |
| 库内已采 | 该号已有 `BloggerPost` | **免费** | 已采过的号复用 |

规则：
- **评估只用「搜索返回 + 列表级」**算指标，够用。
- **已采过的博主**（库里有 active 笔记）→ 直接复用，零调用。
- **详情**只在两处发生：① 用户对某候选点"深看"（抽样几篇，存 `BloggerPost.status='preview'`，不采纳就清理）；② 用户**确认对标后**走正式完整采集。
- 推荐候选池**硬上限 K**（默认 12），每个只列表级。

## 3. 评估内核：四项指标算法（build-ready）

`evaluate(candidate, context)` →

| 指标 | 怎么算 | 依赖账号? | 缺数据时 |
|---|---|---|---|
| **火爆度** popularity | **纯规则，不调 LLM（免费）**：`0.5·follower_score + 0.5·engagement_score`。`follower_score = min(100, 20·log10(follower+1))`（1k≈60 / 1w≈80 / 10w≈100）；`engagement_score` 按近期笔记点赞中位数 / 粉丝数分档 | 否 | 给粉丝分，互动标"未知" |
| **方向契合** relevance | **LLM**：候选简介 + 近期标题 vs `intent`(+ 账号真实赛道) → 0-100 + 一句理由 | 增强 | 只对 intent |
| **可对标性** learnability | **LLM**：方法论是否清晰可提炼 / 是否高度依赖个人资源·颜值·明星身份（难复制）/ 更新是否活跃 → 0-100 + 理由 | 否 | — |
| **可迁移度** transferability | **LLM，仅当 `my_account_id` 在场**：候选打法对"我账号现状"是否够得着、形式匹配 → 0-100 | 是 | 不给该项 |

- **综合分** = 加权（默认 `relevance 0.4 / learnability 0.25 / popularity 0.25 / transferability 0.1`，权重入 `config` 可调；无 transferability 时归一化到前三项）。
- **活跃度** activity 作为 gate：最近一篇 > 60 天 → 标"不活跃"并扣综合分。
- LLM 三项**一次调用批量评一个候选**（合成一个 prompt 出 JSON），省 token。

**Prompt 草图（评估一个候选，一次出 JSON）**：
```
用户想做：{niche}；受众：{audience}；目标：{goal}。
[若有账号] 用户账号现状画像：{my_profile}
候选博主：{name}｜简介：{bio}｜粉丝：{follower}｜近期标题：{titles[:10]}
请输出 JSON：
  relevance:{score,reason}      // 方向是否契合用户想做的
  learnability:{score,reason}   // 是否适合"照着学"(方法论清晰、非纯资源/颜值型)
  [transferability:{score}]     // 仅有账号画像时：对该账号是否够得着、可迁移
一句话中文理由，给小白看得懂。
```

## 4. 三个入口的完整流程

### 入口 A · 关键词搜索（增强已实现）
- **现状**：`GET /bloggers/search` 返回候选，无评分。
- **增强**：① 列表每个候选直接显示**火爆度小标**（纯数据，免费，不调 LLM）；② 每个候选一个**「评估」按钮** → 点了才对**该候选**跑内核（含 LLM）。**绝不对整页搜索结果自动跑 LLM**（烧钱）。
- **出口**：评估满意 → "采纳为对标" → 入库 + 采集。

### 入口 B · 智能推荐（新，异步任务）
```
填 BenchmarkContext 表单
  → LLM 把 niche/intent 扩展成 3-5 个搜索词
  → 并发调 search → 去重合并候选池(截到 K=12)
  → 每个候选取列表级数据（已采过的复用库内）
  → 内核并发打分 → 按综合分排序
  → 返回 Top N 候选卡（四项指标条 + 一句依据）
  → 用户勾选确认 → 入库 benchmark → 触发完整采集
```
- 多次 LLM 调用 → **异步任务**，复用现有 task 队列 + LiveProgress 时间线。
- 结果存 `BenchmarkRecommendationRun`（§5），可回看、可刷新。

### 入口 C · 单博主评分（新，同步 / 快异步）
```
一个候选（搜索结果里选中 / 或粘贴主页链接解析，复用现有 URL 解析）
  → 列表级取数（已采过则复用库内）
  → 内核打分
  → 返回单卡："适不适合当你的对标" + 综合分 + 四项指标 + 依据
  → 用户决定是否采纳为对标
```
- 单候选、单次 LLM → 可同步返回（带 loading），或轻异步。

## 5. 数据模型 / Schema

**复用**：`BloggerProfile`(account_type='benchmark')、`BloggerSearchResult`、`search_bloggers()`、URL 解析、`build_account_content()`。

**新增**：
- `BenchmarkRecommendationRun`（推荐任务，便于异步 / 回看）：
  `id, tenant_id, platform, intent_json, my_account_id?, status, candidates_json(候选 + 各分 + 依据), task_id, created_at`
- 评分（入口C）：单次结果可不落库，随响应返回；如需历史，复用同表 `kind='evaluate'`。
- `BloggerPost.status` 增 **`preview`** 态（仅"深看"抽样详情用）+ 清理任务（按 tenant + 时效）。**入口 B/C 默认不抓详情，preview 是可选增量，不阻塞主流程。**

**Schema（pydantic）**：`BenchmarkIntent`、`BenchmarkContextIn`、`CandidateScore{popularity, relevance, learnability, transferability?, overall, reasons, active}`、`RecommendRequest / RunRead`、`EvaluateRequest / Result`。

## 8. 前端

- **入口位置**：创建对标博主处（"博主资产"右上角"加对标" 或 "数据采集"前置），用 **三段 tab**：`搜索 | 智能推荐 | 链接评分`。
- **智能推荐**：意图表单（领域必填 + 受众/目标/形式 + "我的账号"选择器）→ 提交 → LiveProgress → **候选卡列表**（每卡：头像/昵称/粉丝 + 四项指标条 + 一句依据 + 「采用」）。
- **评分**：粘链接 / 选候选 → 单张大卡（综合分 + 四项 + 依据 + 「采用为对标」）。
- **搜索**：现列表 + 火爆度小标 + 每行「评估」。

## 9. 成本估算（每次操作）

| 入口 | TikHub | LLM |
|---|---|---|
| 搜索 | 1 次搜索（含返回） | 0（火爆度纯数据） |
| 搜索 → 评估某候选 | 1 次列表（复用则 0） | 1 次（评一个候选） |
| 智能推荐 | 1 querygen(LLM) + ~3-5 搜索 + ≤12 列表（复用的减免） | querygen 1 + 候选 ≤12 |
| 单博主评分 | 1 列表（复用则 0） | 1 |
- 推荐的 LLM 调用量随候选池线性 → **K 上限**是成本闸；列表级取数避开小红书详情贵价。

---

## 10. 验收

- 无账号：纯意图能跑出推荐 Top N，每个有四项分（无可迁移度）+ 依据；
- 有多账号：必须选一个，选后多出可迁移度、方向契合更贴该账号；
- 已采过的号评估**零新增 TikHub 调用**；
- 搜索页不点评估则**不产生 LLM 费用**；
- 推荐候选不超过 K；不活跃号被标记降权；
- 采用候选 → 正确入库为 benchmark 并能进入采集。

---

## 11. 不做的事（本方案范围内）

- 不做采集/蒸馏/创作/诊断（确认对标后才进入，单独流程）。
- 不在推荐阶段引入"vs 我的账号"的逐维度深比（那是 ⑤ 诊断）。
- 评估阶段不批量抓详情（成本红线）。
- 不做对标博主的长期监控/演化重评（后续迭代）。
