# 蒸馏产出重定义 · 方案（概念地基）

> 关联：[[蒸馏证据供给优化_方案设计]]（上游 · 已实施：让蒸馏"看得全"）、[[Skill能力包升级_方案设计]]（下游 · 如何投影消费，部分被本文重构）、[[蒸馏车道化_方案]]、[[视觉层_图片理解_方案设计]]。
> 本文是**概念地基**：回答"蒸馏到底该产出什么、各产物归谁、怎么被消费"。**不写代码，落地待排期。**

## 一、背景：现在的蒸馏产物是"四不像"

现在一次蒸馏产出**一份大而全的 `skill_markdown`**（认知/策略/内容/禁区/选题/局限性全塞一坨），创作时被**整段截断塞进 prompt**（`guide.py` 12000 字）。症状：

1. **混了不同用途的内容**：发布节奏、运营习惯、成长趋势这些"认识博主"的分析料，和"怎么写一篇"的创作料揉在一起。创作时前者是纯噪音。
2. **混了不同数据源要求的内容**：发布节奏、成长趋势是**账号级事实**，却从"蒸馏选的那批笔记（快照子集）"算出来——**抽样必失真**（挑了高赞爆款 50 篇算"48 天更新一次"，不是博主真实节奏）。这是一个数据正确性 bug。
3. **把"原料"当"成品"用**：整份画像直接灌进创作，模型要自己反推套用。

根因一句话：**蒸馏产出的其实是"博主画像"（原料），却被当成"创作 skill"（成品）直接消费。**

## 二、核心重定义

**蒸馏的唯一职责 = 从一批笔记（快照）里提炼「可迁移的创作方法」，产出一份「创作画像」。** 它不算账号统计、不写运营诊断、不固化选题——那些各有各的家。

### 术语表（命名是此前困惑的根源，先钉死）

| 术语 | 含义 | 数据源 | 基数 |
|---|---|---|---|
| **博主档案** (Dossier) | 账号级客观事实（节奏/成长/数据面板） | 全量笔记 / 主页元数据 | 一博主**一个**（唯一，刷新） |
| **创作画像** (Profile) | 从一批笔记提炼的、可迁移的创作方法 —— **即现在误名为 "skill" 的东西** | 快照（选的笔记） | 一博主**多个**（命名并存） |
| **投影** (Projection) | 从画像/档案里为某个下游功能提取 + 加工出的成品 | — | — |
| **skill** | 画像在"创作"维度的投影（操作手册），**≠ 画像本身** | — | — |

## 三、两条判据（决定每条信息归哪）

- **判据一（归属）**：「这条，能不能变成'写的时候照着做'的指令？」
  - 能（"标题用 [数字]+[结果]"）→ **创作**
  - 不能，只是"这博主的统计特征/习惯/节奏"→ **档案 / 诊断**
- **判据二（数据源）**：「这条，是从样本能提炼的『方法』，还是需要全量才准的『账号事实』？」
  - 方法（标题公式/结构/语言 DNA）→ **快照蒸**
  - 事实（发布节奏/成长趋势/藏赞比）→ **全量算**（快照会失真）

## 四、蒸馏产出：创作画像（基于快照，可多个）

只装"从样本能提炼 + 服务于做内容"的东西，三段：

### A. 写法层（按车道：图文 / 口播 / 非口播）
标题公式（填空模板）· 开头模板 · 正文/口播结构骨架 · 情绪节奏 · 语言 DNA · CTA 写法 · 封面文案 · 图内信息编排 · 标签策略 · **对比示例（正例/反例）**
→ 消费者：**创作 skill**（怎么写一篇）

### B. 角度 / 立场层
观点张力（选题角度金矿）· 价值立场 · 蹭热点方式 · **薄人设声音**（自称/语气/口头禅）
→ 消费者：**选题器**（现场生成选题）+ 创作借薄人设声音

### C. 创作边界
创作禁区 → 创作与选题都遵守

## 五、蒸馏不产出（剥离出去）

| 剥离 | 为什么 | 新家 |
|---|---|---|
| 发布节奏、成长趋势 | 账号事实，快照**算不准**，需全量 | 博主档案（全量）→ 诊断 |
| 藏赞比、模态对比、总均赞、数据面板 | 账号统计事实 | 博主档案 → 诊断 |
| 思维模式（她的分析工作法） | 是"分析她"，不是"教你写" | 诊断 |
| 固化的 8 个选题灵感 | 会过时、不匹配用户当下意图 | 选题器用 B 层 + 当下热点**现场生成** |

## 六、身份与基数

- **创作画像可以多个**：一博主底下 (高赞爆款笔记→**爆款画像**)、(某赛道笔记→**垂类画像**)、(近 3 个月→**近期画像**) 是不同侧面，命名并存。身份坐标 = (博主 + 快照 + 意图)。
- **博主档案只有一个**：真实发布节奏是账号客观属性，跟你选哪批笔记无关，全量算一次，所有画像共享。
- **两种"重蒸"要分开**：
  - **版本迭代**（同快照、同意图，蒸馏逻辑改进后重蒸）→ 新版**覆盖**旧版（现有 `archive_active_skills` 逻辑对这种情况是对的）。
  - **新画像**（换了快照/意图）→ 应**并存**，不该 archive。现状一刀切 archive，**丢失了多画像**——这是要改的点。

## 七、三个下游怎么消费

| 下游 | 系统现状 | 取画像/档案的哪部分 |
|---|---|---|
| **创作 skill** | `generate_package_content` | 画像的 [写法层 + 薄人设]，按内容类型只取对应车道 |
| **选题器** | `generate_xhs_topic_ideas` | 画像的 [角度/立场层] + 当下热点，现场生成（不复述固化选题） |
| **诊断** | `appraisal` / `account_audit` | 博主档案（全量事实），与画像无关 |

## 八、落地要动的三件事（待排期，本文不含实现）

1. **拆产线（含修数据 bug）**：`analysis.analyze_posts` 里的账号事实统计（`analyze_posting_frequency` / `analyze_growth_trend` / 藏赞比 / 模态对比）从蒸馏移出，改由「博主档案」从**全量 posts** 算。→ 修了"发布节奏从快照算不准"的 bug。
2. **蒸馏瘦身**：蒸馏产出收窄成"创作画像"（A/B/C 三段）；`artifacts` 不再把账号事实 / 固化选题渲染进创作产物。
3. **画像正名 + 多画像**：`BloggerSkill` 概念上是"创作画像"，给身份/命名；松开"单一 active"，创作/选题时**选用哪个画像**；区分版本迭代（覆盖）与新画像（并存）。

## 九、与已有方案的关系

- **[[蒸馏证据供给优化_方案设计]]（已实施）**：让蒸馏"看得全"（图内文字进模型 22%→72% 等）。**正交，本文不影响它**——喂料更全，产出仍按本文收窄。
- **[[Skill能力包升级_方案设计]]（部分重构）**：其"按车道加载""模板 + 正反例"仍然有效，但**框架从"把 skill 拆开"改成"从创作画像投影出 skill"**。即：能力包升级 = 本文"创作 skill 投影"这一步的具体做法。
- **本文 = 地基**：先定"产出什么、归谁"，再谈"怎么投影、怎么提质"。

## 十、未来方向：成长轨迹分析（暂不实施，先存档）

用户提出的高价值想法：把博主切成**低谷期（发得多不涨）→ 爆火期 → 平稳期**，用于 ① 定"该蒸哪批笔记"（爆火期笔记 → 爆款画像选材）② 蒸"她做对了什么才火的"（转折点前后内容对比 = 成功转折方法论）。

**数据现实（已查证）**：
- 小红书**不对外公开**历史涨粉曲线（博主自己后台可见近 7/30 日）。
- **TikHub 小红书无趋势接口**（「达人趋势对比」仅支持抖音）；只给当前粉丝数（单点）+ 笔记列表（互动 + 时间）。
- 千瓜 / 新红有历史涨粉曲线，但靠自己每日快照攒，需付费对接。

**可行替代（用已有数据）**：用"笔记互动 + 发布时间"近似**内容爆发轨迹**（业内公认"爆文曲线 ≈ 涨粉曲线"）。
- ⚠️ 坑：早期互动低可能是粉丝基数小、非内容差。缓解：看**相对自身基线的跃升** + 量级台阶（百→千→万）+ 发布密度，辅以当前粉丝数做锚。
- 归属：属于**博主档案的时间轴维度**（全量数据），并给**爆款画像**提供选材依据。

**结论**：先不做；要做时首选"内容爆发轨迹（零新数据依赖）"，同时从今往后每次采集记"粉丝数 + 日期"为将来铺路。

## 十一、不做 / 边界

- 本文只定"产出什么"，不含实现代码。
- 不做标准 Agent Skill 包（可移植/导出）——已决定暂不需要，聚焦提质。
- 不动采集 / 视觉 / 证据供给（上游，已优化）。
- 成长轨迹分析暂缓（见第十节）。
