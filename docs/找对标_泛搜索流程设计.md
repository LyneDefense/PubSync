# 「找对标博主」改版 + 泛搜索流程设计

> 状态：**方案定稿，待拍板后实现**。本文件只定「找对标」这一块。
> 关联：[[搜寻对标博主_方案设计]]（旧版，将被本文件取代）、[[效果看板_方案设计]]、[[采集模块设计]]。

## 一、为什么改（现状的问题）

现在「找对标博主」有三个入口（智能推荐 / 关键词搜索 / 链接评分），问题：
1. **找人和评人混在一起**：智能推荐对每个候选都调 LLM 打分，又慢又贵；而"评分"其实是另一件事。
2. **召回太浅、漏真·对标**：候选只来自「关键词→平台用户搜索、第 1 页、截 12 个」。而平台用户搜索是**昵称/简介的文本匹配 + 平台排序**——像「保险经纪人娜姐」这种**人名号**，搜"香港保险"根本排不上来（真实踩坑案例）。
3. **链接评分放错地方**：它是"给某个博主打分"，不属于"找对标"。

> 已确认的平台能力（小红书，TikHub）：`search_users`（关键词文本搜，我们只传 keyword+翻页，无领域分类/排序参数）、`search_notes`（关键词搜笔记）、**用户关注列表 following**、用户粉丝列表 fans、用户信息、用户笔记、笔记详情。**没有"相似博主/推荐账号"这种账号级推荐接口**；"首页推荐"是内容流、不针对领域，无用。

## 二、改版后的结构

**「找对标博主」拆成两个入口，都只负责"找到人 + 放进对标库"，一个 LLM 评分都不做：**

1. **精确搜索**：按关键词 / 博主姓名搜 → 平台返回账号列表 → 直接「采用」入对标库。**不评价、不打分。** （= 现在的关键词搜索，保留）
2. **泛搜索**：填方向，系统帮你"广撒网"扒出一批可能合适的对标号（见第三节，核心）。

**删掉「链接评分」。** "给对标博主打分（值不值得学）"搬到**对标库**：每个对标博主一个「评估」按钮，按需才算。
> ⚠️ 这与现有的「对标诊断」(account_audit) 不是一回事：对标诊断是**拿你的账号 × 对标博主比、诊断你的差距**；新的"评估"是**单独判这个博主本身值不值得学**。两者并存，不合并。

**tab 改名**：从「找对标博主」改成 **「选对标」**（短、名词、覆盖"找+挑"；候选 alt：「找对标」）。

## 三、泛搜索流程（核心，逐步写清）

### 总览（一张图）

```
①填领域(多个,必填)  ②选我的账号当种子(可选)
            │
            ▼
③LLM 扩展方向 → 用户勾选 + 调权重(系统给默认)
            │
            ▼
④三路召回(见 3.4) → 候选账号(带客观理由,不逐个调LLM)
            │
   用户对候选三选一:
   ├─「采用为对标」 → 进【入选篮子】(暂存,不立即入库)
   ├─「设为种子,继续找」→ 进【种子托盘】→ 回到④(以种子找相似)
   └─「跳过」
            │
            ▼ (觉得够了,点"完成")
⑤确认【入选篮子】→ 批量入对标库 → 顺势引导去采集/蒸馏
```

每一轮④都**自动排除**已展示 / 已入选 / 已在对标库的号，所以候选会越挖越少——这是循环的天然刹车。

### 3.1 步骤一：填领域（必填，可多个）

- 输入：标签式（chips）。用户可填多个领域词，如 `香港保险` `保险经纪` `储蓄险`。
- 多领域 = 召回时**并集取数**（每个领域都会去搜），扩大覆盖。
- 至少 1 个，否则不能开始。

### 3.2 步骤二：选种子账号（可选）

- 若用户已录入「我的账号」，可选其中一个当**种子**：让系统"找和它像的号"。
- 种子也可以来自后面④的候选（见 3.5「设为种子」）。
- 种子放进一个**可增删的「种子托盘」**（见 3.6），**上限 3 个**（太多会稀释"相似"信号）。
- 没有种子也能跑——纯靠领域召回。

### 3.3 步骤三：LLM 扩展方向 + 用户选/调权重

- LLM 基于「你的领域(+种子的内容指纹)」产出 **6–10 个扩展方向**，每个带：方向名 + **默认权重(0–100)** + 一句话理由。默认权重 = LLM 判断"这个方向离你目标多近"。
- UI（chips + 权重，主操作是勾选）：

```
你的方向：[香港保险 ✕] [储蓄险 ✕]   ＋自己加

AI 扩展（勾选要搜的，拖动调权重）：
☑ 储蓄险科普         ▉▉▉▉▉▉▉░░ 78   "和你最贴近"
☑ 香港身份/移民理财   ▉▉▉▉▉░░░░ 52
☑ 保险避坑/拒赔       ▉▉▉▉░░░░░ 41
☐ 家庭资产配置         ▉▉▉░░░░░░ 30
☐ 基金/股票            ▉▉░░░░░░░ 18   "偏远,默认不选"
                                         [开始找对标 →]
```

- 默认勾选 top 3–4；用户可勾/取消、拖滑条调权重、也能自己加方向。不想动就直接开始（用系统默认）。
- **权重的作用**（一句话告诉用户：「权重高的方向，会被搜得更深、候选里也更靠前」）：
  - ① **分配召回配额**：高权重方向翻更多页 / 给更多候选名额；
  - ② **候选排序加分**：命中高权重方向的候选更靠前。

### 3.4 步骤四：三路召回 → 候选账号

平台用户搜索单条腿召回不全（漏人名号），所以**三路合并去重**：

| 召回路 | 怎么做 | 解决什么 |
|---|---|---|
| **A. 搜索用户** | 每个选中方向 → `search_users`(翻深几页) | 名字/简介含词的号 |
| **B. 搜索笔记→取作者** | 每个方向 → `search_notes` → 取爆款笔记的**作者**去重 | **捞到"靠内容出圈、但名字搜不到"的人名号**(娜姐就属这类) |
| **C. 种子的关注列表** | 有种子时 → 拉种子 `following` → 同行互关的号 | "找相似",更多同类 |

- 三路结果**按 external_id 去重合并**，按配额（权重）截断成候选池（默认上限 30，可配）。
- 每个候选给**客观理由**（不逐个调 LLM，便宜）：命中方向标签 + 粉丝数 + 发帖量 + 活跃度(最近X天更新) + 火爆度(粉丝+互动,规则算) + 个人号/机构号。
  - 例：`命中「储蓄险科普」· 1.2万粉 · 86篇 · 近5天更新 · 互动率~3% · 个人号`
  - 可选增强：**一次批量 LLM**（一个 prompt 评所有候选）给"匹配度"标签 + 一句话，控成本；不做"每候选一次 LLM"。
- 候选按"火爆度 + 命中权重 + 活跃 + 个人号优先 + (有种子时)越像种子越靠前"的**规则综合分**排序。

### 3.5 步骤五：候选三选一

每张候选卡两个动作（外加跳过）：
- **采用为对标** → 进【入选篮子】（暂存，最后统一入库）；
- **设为种子，继续找** → 进【种子托盘】→ 触发新一轮④（以托盘里的种子找相似）。
- 一个候选可"既采用、又顺手设为种子再挖一批同类"。

### 3.6 步骤六：篮子结账（闭环收口）

- 跨多轮选中的"对标"都先进**【入选篮子】**（购物车），用户随时能看、能移除。
- 点**「完成 / 入库」** → 篮子里的号**批量入对标库** → 自然引导下一步（去采集 / 蒸馏）。
- 找不到也能空手退出，不强迫入库。

## 四、循环的边界规则（防打转、防卡死）

1. **种子托盘**：可增删，**上限 3**；加第 4 个时提示"先移除一个"。每轮用托盘当下的种子。
2. **去重记忆**：整个发现会话维护"已展示 / 已入选 / 已在库"集合，每轮只给**新**候选。
3. **挖干刹车**：某轮新候选 < 3（或 0）→ 提示「这批种子/方向能挖的差不多了，要不 ①换/加领域 ②换种子 ③去看篮子结账」。
4. **安全上限**：单次会话 ~10 轮兜底（防跑飞/烧钱），正常到不了。
5. **出口阶梯（"一直找不到"时）**——每轮结果区顶部常驻这些非循环出口：
   - 先**解释为什么没货**（领域太窄/太宽 / 被活跃度·个人号筛掉 / 平台本来就少）；
   - **改领域**（放宽或具体化）、**换/清种子**、**放宽筛选再看一批**、**继续翻深**；
   - **精确搜索兜底**：知道名字就直接去入口①按名字搜；
   - **诚实兜底**：这赛道确实没几个就直说，建议相邻方向，不硬凑水号。

## 五、种子「找相似」到底怎么实现（基于已确认接口）

小红书**没有**"相似账号"接口，所以"找相似"用两个**真实**杠杆近似：

1. **关注列表挖掘（主）**：拉种子的 `following`（接口已确认存在）。同行/同类创作者**往往互相关注**，这是小红书上最靠谱的"找相似"。
   - 杂质过滤：following 里有朋友/明星/品牌官号 → **必须过一遍筛子**（同领域 + 个人号 + 够活跃 + 粉丝量级合理）才进候选。
   - 截断：大号关注几百个 → 按"和种子/领域的匹配度"排序，只取 top。
2. **内容指纹重排（辅）**：种子若已采过笔记，用其领域/标签/高频选题/标题做指纹，给候选"越像种子越靠前"。指纹也反哺③的扩词。

> 抖音：有自己的用户搜索；"关注列表/相似"能力未逐一确认。**MVP 以小红书为准**，抖音先复用 A/B 两路召回（用户搜索 + 笔记取作者），关注列表挖掘待确认接口后补齐。

## 六、成本

- 每轮：**1 次 LLM**（扩词，且只在第一轮/改方向时）+ 可选 **1 次批量 LLM**（候选匹配标签）+ N 次搜索/关注列表调用。
- 比旧版"每候选 1 次 LLM"便宜一个量级；评分（贵）挪到对标库按需触发。

## 七、数据模型 / 状态

- 新增 **`BenchmarkDiscoverySession`**（或扩展现有 `BenchmarkRecommendationRun`）持久化一次发现会话：领域、扩展方向(+权重+选中)、种子托盘、各轮候选、已展示集合、入选篮子、状态。**刷新/切走不丢**（这是个多步可迭代向导）。
- 入库沿用现有"采用→创建 BloggerProfile(account_type=benchmark)"。

## 八、关键文件（实现时）

- **后端**：
  - `tikhub_client/`：加 `get_user_following`（关注列表）适配 + `search_notes`→作者 的适配（B 路）。
  - `benchmark_discovery/querygen.py`：多领域 + 种子指纹的扩词，产出"方向+默认权重"。
  - `benchmark_discovery/recall.py`（新）：三路召回 + 合并去重 + 配额。
  - `benchmark_discovery/engine.py`：从"每候选 LLM 打分"改成**规则综合分**排序（保留 popularity，去掉逐个 LLM；批量匹配标签可选）。
  - `benchmark_discovery/service.py`：会话/轮次编排、种子托盘、去重、挖干判定。
  - `models/`：`BenchmarkDiscoverySession`；`schemas/`、`api/bloggers.py`：会话相关端点（开始/下一轮/设种子/采用/结账）。
  - 评分搬迁：对标库"评估此博主"端点（复用旧 `evaluate_candidate` 的维度）。
  - `config.py` + admin：召回配额、候选上限、种子上限、挖干阈值、权重默认。
- **前端**：
  - `FindBenchmarkView.vue`：改成两入口（精确搜索 / 泛搜索）。

  - 对标库页：每个博主加「评估」。
  - store / api / types：会话状态 + 端点。

## 九、分阶段

- **Phase 1（先落地、最快见效）**：
  - 拆两入口、删链接评分、评分搬到对标库；
  - 泛搜索**单轮**：多领域 → 扩词选权重 → **A+B 两路召回**（用户搜索 + 笔记取作者，先不做关注列表）→ 候选带客观理由 → 入选篮子 → 结账；
  - 种子先用"内容指纹重排"近似。
- **Phase 2（把"找相似"做真）**：
  - **C 路关注列表挖掘** + 多轮循环 + 种子托盘 + 挖干提示 + 出口阶梯 + 会话持久化。

## 十、验证

- 后端单测：多领域扩词；三路召回合并去重；规则综合分排序；关注列表筛子（杂质过滤+截断）；挖干判定；会话去重不重复给。
- 真机：填"香港保险" → 泛搜索能否把"娜姐类"人名号捞出来（重点验 B 路笔记取作者）；设娜姐为种子 → 关注列表能否带出更多同类；多轮不重复；找不到时出口可用；精确搜索按名字能直达。
- 按 CLAUDE.md：ruff + pytest + build → 提交(无 co-author) + push → 部署 → 真机走查。

## 十一、不做的事

- 不做"每候选一次 LLM"的重打分（评分挪到对标库按需）。
- 不用"首页推荐"（不针对领域）。
- 不在"找对标"里做账号诊断（那是对标诊断）。
- 抖音的关注列表挖掘待接口确认前不强上。
