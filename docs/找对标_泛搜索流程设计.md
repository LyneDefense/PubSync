>
> 状态：~~已实现并上线（2026-06）~~ → 待移除。本文件描述**曾经的真实实现**，并记录关键演进/踩坑，避免后人重走弯路。
> 关联：[[搜寻对标博主_方案设计]]（最初版，已废）、[[效果看板_方案设计]]、[[采集模块设计]]。
> 一句话：**泛搜索是个漏斗**——领域 → AI 帮你把范围收窄到几个细分角度 → 广召回一批 → **逐个核验账号本身**（不是看一条笔记）→ 只留相关的、按综合分排 → 你「采用/不要」→ 保存进对标库。

## 一、为什么是这套（演进 + 踩过的坑）

这块改了好几版，每版都是被真实问题逼出来的，记下来免得回退：

1. **最初三入口**（智能推荐 / 关键词搜索 / 链接评分）的问题：找人和评人混在一起、每候选调一次 LLM 又慢又贵、召回浅漏掉"人名号"。
2. **第一版重构**（角度 + 三列工作台 + 种子）上线后，用户两条反馈直接否掉了"种子"：
   - **没有"收窄"的感觉** —— 像"搜一堆 → 自己整理进桶",不像漏斗。→ 把收窄做到**角度**这一层（AI 迭代推细分领域，用户选够 N 个）。
   - **"设为种子"不成立** —— 用户根本没法一眼判断谁适合当种子；他要是认得某个号好，直接就"收为对标"了。→ **彻底砍掉种子概念。**
3. **关键平台事实（实测确认）**：**TikHub 小红书没有"关注列表 / 粉丝列表 / 相似账号"任何社交图谱接口**（把全部 60 个 xhs 端点列出来确认过；网上搜到的"获取用户关注列表"文档其实是微博/抖音的）。→ 种子"找相似"靠 following 这条路**根本走不通**，这也是砍种子的硬理由。"找相似"改成**基于已有对标博主的存量笔记内容**找同类，并独立成一个子 tab。
4. **召回精度问题**（用户实测）：`董保险避坑指南`、`不普通人类实录` 这类号混进来——它们只是**写过一条沾边的笔记**，账号本身根本不做香港保险。根因：**"写过一条匹配笔记的作者"是内容级信号，不是账号级**；光看昵称/一条笔记判不出账号定位。→ 加 **"账号核验"** 阶段：像人一样点进主页看简介+最近发什么，再判相关度。
5. 其它实现坑（已修，见第八节）：B 路作者藏在 `noteCard.user`（camelCase）；方向权重原来被 log 压到形同虚设；粉丝数把"小红书号"当数字（94351549807）。

## 二、最终结构：「选对标」= 3 个子 tab

都只负责"找到人 + 放进对标库"，**不在这里做"值不值得学"的重打分**（那是对标库的事 / 对标诊断的事）。

1. **精确搜索**：关键词/姓名 → `search_users` → 列表 → 直接「采用」入库。不评分。
2. **泛搜索**（核心，见三）：不知道学谁时，漏斗式收窄到几个好号。
3. **找相似**（见四）：已经有几个好对标，从对标库挑参照 → 找更多同类。

> 「链接评分」已删。"单独判某个博主值不值得学"可后续在对标库做一个「评估」按钮（与 account_audit「对标诊断=你 vs 对标」不同），目前未做。

## 三、泛搜索流程（漏斗，确定性控制器 + AI 只在节点内填内容）

```
①填领域(可多个)
   └─[下一步]→ 后端 stage = choose_angles
②选细分角度(收窄)  ◀── agent 迭代推荐
   AI 推 N 个细分角度(chips) → 用户 选/×排除/「再推几个」
   建议选 ~3 个(target);选多了提醒但允许;推角度有轮数上限+短超时+规则兜底
   └─[就用选中的开始找]→ stage = workspace,触发召回(异步任务)
③候选工作台(两列:候选 | 已选)
   候选卡:硬数据+软数据,点「看依据」展开
   每张卡 [采用]→已选 / [不要]→剔除(进 seen 不再出现)
   [再找一批] = 同角度再来一轮(去重、追加、新结果置顶)
   [清空候选]
   已选列 [保存到对标库](幂等,可随时存、增量存) / 每项可[移除]
```

- **会话两阶段**：`choose_angles`（选角度）→ `workspace`（看候选）。`source = broad`。
- **等待发生在 HTTP 边界**：每步是一次普通 request/response，服务端不挂进程；"用户不输入"不卡 worker。只靠会话 `expires_at`（默认 2h）+ 定时清理过期。
- **召回是异步任务**（打 TikHub 多次，耗时），进度走 LiveProgress（逐角度 + 核验进度事件）。其余操作（选角度/采用/移除/保存）都是同步轻量。

### 3.1 角度收窄（把"缩小范围"做实）

`querygen.suggest_subniches(domains, selected, rejected, shown, n)`：把"已选/已拒/已展示"喂回 agent，让它**只出新的、更贴近已选调性、避开已拒**的细分角度。护栏（缺一不可）：
- **软目标 target(=3)**：建议选 3 个左右；任何时候都能「就用选中的开始找」，不强制；
- **轮数上限 max_rounds(=4)**：连推几轮还没凑够就停，提示用现有的搜（防空转/越推越水）；
- **>target 容忍**：提醒"选多了会搜得杂"，但允许；
- **短超时 + 规则兜底**：推角度的 LLM 调用套 45s 硬超时（`_json_with_timeout`），超时/失败 → 规则兜底（领域 + 常见角度后缀）。

### 3.2 三段式召回（核心精度保障）

`flow.run_recall` 按选中角度跑三段：

| 段 | 做什么 | 解决 |
|---|---|---|
| **① 广召回**（便宜） | 每个选中角度跑 **A 搜用户**(`search_users`,权重深翻页) + **B 搜笔记取作者**(`search_notes`→`noteCard.user`)，交错去重，先拉 `prevet_cap(15)` 个 | 把"靠内容出圈、名字搜不到的人名号"也捞进来（娜姐/达尔文-重疾险这类，全靠 B 路） |
| **② 账号核验**（贵,但只对要展示的核） | 对这 15 个**逐个补取主页**（`user_info`：简介/IP/粉丝/获赞与收藏）+ **近期笔记标题**（`user_notes`），并发 `vet_concurrency(5)`;再 **一次 LLM 批量**判每个账号"本身是不是持续在做这个领域"(0–100 相关度) | **内容级 → 账号级**:剔掉"只写过一条沾边、主体是生活/日常"的泛号 |
| **③ 筛选 + 排序** | 相关度 < `relevance_floor(30)` 直接剔；其余按**综合分**降序，取 `candidate_cap(12)`，**追加**到候选池（新结果置顶，`pool_cap(60)` 封顶） | 只把相关的、好的露出来 |

- 看过的（含被剔的）都记进 `seen`，下轮「再找一批」不重复召回/核验。
- **相关度判定**（`vetting._llm_relevance`）：一次 LLM 看所有候选的「昵称+简介+IP+近期标题」，判分从严（生活/日常/泛知识类即使偶尔提到也给低分）；**LLM 失败 → 规则兜底**（领域词在简介+标题里的命中数）。
- **核验顺带把硬数据取回来**填卡片（一举两得：核验 + 展示）。也顺带修了"粉丝未知"——核验拿到真实粉丝数。

### 3.3 候选卡：硬数据 + 软数据（点「看依据」展开）

- **硬数据（小红书本身）**：粉丝数、获赞与收藏、简介、IP 属地、近期笔记命中条数(+标题)。
- **软数据（我们算的）**：相关度(LLM 0–100)、火爆度(粉丝量纲 `popularity_score`)、综合分。
- **综合分** = `0.6×相关度 + 0.4×火爆度`，机构号 ×0.9（学打法优先个人号）。**相关度为主**——避免"粉丝多但不对路"的号靠前。

## 四、找相似（独立子模块，基于对标库存量笔记）

- 从**对标库**挑 1+ 个博主当参照（`account_type=benchmark`）。
- `flow.derive_similar_keywords`：取这些博主的 `niche` + 其 `BloggerPost` 的**高频话题标签**（存量笔记，不额外取数）→ 关键词。
- 用这些关键词当"角度"，**直接进 workspace**（无角度收窄阶段，`source=similar`），走**同一套**三段式召回 + 候选工作台。
- 比"关注列表找相似"更干净（关注列表全是朋友/明星/品牌官号噪声），且小红书本就没有关注列表接口。

## 五、鲁棒性（硬要求，处处兜底）

| 环节 | 正常 | 失败/超时 → 兜底 |
|---|---|---|
| 推角度（LLM） | 出带理由的细分角度 | 45s 超时/失败 → 规则后缀兜底 |
| B 路搜笔记取作者 | `noteCard.user` 取作者 | 端点失败 → 静默退化成只用 A 路（搜用户），打日志 |
| 账号核验·取数 | user_info + user_notes | 某字段取不到 → 该字段留空，不影响其它 |
| 相关度判定（LLM） | 批量判 0–100 | 失败 → 规则兜底（领域词命中数）|
| 会话空闲 | 等待在客户端,服务端不挂进程 | `expires_at`(2h) + 定时清理标 `expired`;过期回来友好提示重开 |

最坏情况（LLM 全程不可用）：泛搜索退化成"规则角度 + 纯三路召回 + 规则相关度(命中数)排序",**功能不挂**,只是少了智能扩角度/语义判相关。

## 六、TikHub 端点事实（实测，避免再猜错）

- **B 路 `search_notes`（web_v3 `fetch_search_notes`）**：作者在 `data...notes[].noteCard.user`，字段是 **camelCase**（`userId` / `nickName` / `xsecToken`）。旧解析只认 `note_card`(snake) → 0 作者（踩过）。
- **`get_user_info`（核验用）**：用 `app/get_user_info`（只需 `user_id`）；返回里**简介在 `basicInfo.desc`、IP 在 `ipLocation`、粉丝/获赞收藏在 `interactions` 列表**（`type=fans` / `type=interaction`），不是独立数值字段。注意主池第一个 `app_v2_share` 需 `share_text`，只传 user_id 必 400 → 核验用**精简池**（app 优先、剔掉 share_text 端点）省时。
- **`get_user_notes`（核验用）**：`data.data.notes[].display_title` 取近期标题。
- **没有**：关注列表 / 粉丝列表 / 相似账号 / 账号级推荐。"首页推荐"是内容流、不针对领域，无用。

## 八、成本 / 延迟

- 一轮"找候选" ≈ **广召回 ~30s + 核验 ~30–40s（15 个并发取数 + 1 次批量 LLM）≈ 1 分钟+**。这是刻意的"慢一点换干净"。
- LLM 调用：推角度（每轮 1 次，仅 choose_angles）+ 相关度判定（每次召回 1 次批量）。比"每候选 1 次 LLM"省一个量级。
- TikHub 调用：广召回 N 角度 ×(翻页搜用户 + 搜笔记) + 核验 ~15×2（user_info + user_notes）。核验是大头，故限 `prevet_cap`。

## 九、已知局限 & 可调项

- **相关度阈值 30**：偏松时擦边号（如日常号被 LLM 误判 ~40）可能露出，但火爆度低、排末尾、可「不要」。嫌松可调 40–50。
- **LLM 偶有误判**：仅给昵称+简介+近期标题，个别会判偏；靠"判分从严 prompt + 阈值 + 人工不要"兜。
- **抖音**：核验主要为小红书设计；抖音 `user_notes` 池缺失 → 近期标题取不到，相关度退化成靠昵称+(可得的)简介。抖音精度弱于小红书。
- **延迟**：一轮 1 分钟+，对低频高价值的"找对标"可接受；嫌慢可降 `prevet_cap` 或换更快模型。

## 十、不做的事

- 不做"每候选一次 LLM 重打分"（综合分是规则+一次批量相关度）。
- 不让 AI 掌舵流程：阶段/选项写死，AI 只在节点内填内容（推角度、判相关度），失败即规则兜底。
- 不用关注列表/相似账号接口（小红书没有）；不用"首页推荐"；不在"找对标"里做账号诊断。
- 不重做"种子"概念（已论证不成立）。
