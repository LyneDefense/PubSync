# 「找对标博主」改版 + 泛搜索流程设计

> 状态：**方案定稿，待实现**。本文件只定「找对标」这一块。
> 关联：[[搜寻对标博主_方案设计]]（旧版，将被本文件取代）、[[效果看板_方案设计]]、[[采集模块设计]]。
> 形态结论：**固定流程骨架（状态机）+ 在固定节点给用户结构化选项/输入 + AI 只在节点内填内容、不掌舵流程走向**。这样 AI 乱不了，又保留它的"聪明"，成本也可控（LLM 只在固定节点调）。

---

## 一、为什么改（现状的问题）

现在「找对标博主」有三个入口（智能推荐 / 关键词搜索 / 链接评分），问题：
1. **找人和评人混在一起**：智能推荐对每个候选都调 LLM 打分，又慢又贵；"评分"其实是另一件事。
2. **召回太浅、漏真·对标**：候选只来自「关键词→平台用户搜索、第 1 页、截 12 个」。而平台用户搜索是**昵称/简介的文本匹配 + 平台排序**——像「保险经纪人娜姐」这种**人名号**，搜"香港保险"根本排不上来（真实踩坑）。
3. **链接评分放错地方**：它是"给某个博主打分"，不属于"找对标"。

**已确认的平台能力（小红书，TikHub）**：`search_users`（关键词文本搜，我们只传 keyword+翻页，无领域分类/排序参数）、`search_notes`（关键词搜笔记）、**用户关注列表 following**、用户粉丝列表 fans、用户信息、用户笔记、笔记详情。**没有"相似博主/推荐账号"账号级推荐接口**；"首页推荐"是内容流、不针对领域，无用。

## 二、改版后的结构

**「找对标博主」→ 改名「选对标」，两个入口，都只负责"找到人 + 放进对标库"，不做 LLM 评分：**
1. **精确搜索**：按关键词/姓名搜 → 平台返回账号列表 → 直接「采用」入库。**不评价、不打分。**
2. **泛搜索**：固定流程 + 结构化反馈节点 + AI 填内容（见第三节，核心）。

**删掉「链接评分」。** "给对标博主打分（值不值得学）"搬到**对标库**：每个博主一个「评估」按钮，按需才算。
> ⚠️ 与现有「对标诊断」(account_audit) 不同：对标诊断是**你的账号 × 对标比、诊断你的差距**；新"评估"是**单独判这个博主本身值不值得学**。两者并存，不合并。

## 三、泛搜索 = 固定流程状态机 + 结构化反馈节点 + AI 填内容（核心）

### 3.1 状态机总览

```
S1 意图(表单)
   领域 chips(必填,可多个) + 选种子账号(可选,下拉我的账号)        →[提交]
        │
        ▼
S2 确认方向(AI 产出 → 结构化多选)
   AI 扩展 6–10 个方向(带默认权重) → 前端"可勾选 chips + 权重滑条"
   用户勾选 / 调权重 / 自己加                                     →[提交]
        │
        ▼
S3 召回与候选(工具/AI 干活 → 列表)            ◀───────────────┐
   三路召回 → 候选卡(带客观理由)                               │
   每张卡可 [采用] / [设为种子],可多选                          │
        │                                                      │
        ▼                                                      │
S4 「下一步」网关(结构化选项 + 兜底输入框)                       │
   AI 看这轮结果,给出按钮:                                      │
   ○ 继续找更多(同条件再来一批) ───────────────────────────────┤ (回 S3)
   ○ 以选中的当种子,找更像的 ───────────────────────────────────┤ (回 S3)
   ○ 换/调方向 ───────────────────────────────→ (回 S2)         │
   ○ 我有具体要求… → 弹【输入框】(如"排除机构号/粉丝>1万") ─────┘ (回 S3,带筛选)
   ○ 完成,把选中的入库 →
        │
        ▼
S5 完成 → 【入选篮子】批量入对标库 → 引导去采集/蒸馏
```

主循环 = **S3 ⇄ S4**（带去重、挖干提示、轮数上限）；要换方向回 S2。

### 3.2 逐节点说明

- **S1 意图**：领域 chips（必填，可多个，并集召回）；可选从「我的账号」选一个当**初始种子**。
- **S2 确认方向**：AI 基于「领域(+种子内容指纹)」产出 6–10 个扩展方向，每个带**默认权重(0–100)**（=AI 判断离你目标多近）+ 一句理由。前端渲染成**可勾选 chips + 权重滑条**，默认勾 top 3–4；用户可勾/调/加，也可不动直接开始。
  - 权重作用（告诉用户：「权重高 → 搜得更深、候选更靠前」）：① 分配召回配额；② 候选排序加分。
- **S3 召回与候选**：跑三路召回（见第四节）→ 候选卡（带客观理由，不逐个调 LLM）→ 每张可 [采用]/[设为种子]，可多选。自动排除已展示/已入选/已在库的号。
- **S4 下一步网关**：AI 看这轮结果，生成**结构化选项**（上面那几个按钮）+ 一句小结（"第2、5个像种子；前3个偏机构号"）+ 一个**兜底输入框**接长尾要求。用户的选择决定跳转。
- **S5 完成**：入选篮子批量入对标库，顺势引导下一步。

### 3.3 统一「待办」契约（server-driven，前端通用渲染）

每次流程**暂停等用户**时，后端返回一个结构化"待办"，前端按字段通用渲染，**不用为每个节点写死 UI**：

```jsonc
// 后端 → 前端：本节点要用户做什么
{
  "session_id": "...",
  "stage": "intent | confirm_directions | review_candidates | next_step | done",
  "message": "这轮找到 8 个;第2、5个和娜姐很像,前3个偏机构号",
  "options":    [{ "id": "more", "label": "继续找更多" }, ...],   // 渲染按钮/多选
  "directions": [{ "id","label","weight","reason","selected" }], // S2: chips+权重
  "candidates": [{ ...带理由的候选, "actions": ["adopt","seed"] }], // S3: 候选卡
  "input":      { "key":"refine", "placeholder":"补充要求,如 排除机构号" }, // 兜底
  "basket":     [...已入选], "seeds": [...种子托盘], "round": 2
}
```

```jsonc
// 前端 → 后端：用户的结构化回应
{
  "session_id": "...", "stage": "review_candidates",
  "picked_adopt": ["uid1","uid2"],   // 采用进篮子
  "picked_seed":  ["uid5"],          // 设为种子
  "directions":   [{ "id","selected","weight" }],  // S2 用
  "choice":       "seed_more",       // S4 选的按钮
  "text":         "只要个人号"        // 输入框(可空)
}
```

后端据此推进到下一节点，再返回下一个"待办"。**流程图写死，AI 只填 message/directions/candidates/options 的内容。**

### 3.4 谁干活、谁做主

| 谁 | 负责 |
|---|---|
| **确定性流程控制器** | 流程图、节点转场、三路召回的"全跑"、去重、轮数/预算上限、篮子收口 —— **写死，AI 改不了** |
| **AI（仅节点内）** | S2 扩方向+权重；S3 判候选匹配/生成理由（可批量一次）；S4 生成"下一步"选项与小结；把 S4 输入框的话翻成筛选/排序条件 |

成本可控：LLM 只在 S2/S3/S4 各调一次（S3 可批量评所有候选），**不是每微步都调**。

## 四、三路召回（解决"漏人名号"，确定性全跑）

平台用户搜索单条腿召回不全，所以**三路合并去重**（这是确定性的，保证都跑）：

| 路 | 怎么做 | 解决什么 |
|---|---|---|
| **A 搜索用户** | 每个选中方向 → `search_users`(翻深几页) | 名字/简介含词的号 |
| **B 搜索笔记→取作者** | 每个方向 → `search_notes` → 取爆款笔记的**作者**去重 | **靠内容出圈、名字搜不到的人名号**(娜姐就属这类) |
| **C 种子关注列表** | 有种子时 → 拉种子 `following` → 同行互关的号 | "找相似",更多同类 |

三路按 external_id 去重合并 → 按配额(权重)截断（默认上限 30，可配）→ 规则综合分排序。

## 五、种子「找相似」怎么实现（基于已确认接口）

小红书**无"相似账号"接口**，用两个真实杠杆近似：
1. **关注列表挖掘（主）**：拉种子 `following`（接口已确认存在）。同行往往互相关注 → 最靠谱的"找相似"。
   - **杂质过滤**：following 里有朋友/明星/品牌官号 → 必须过筛子（同领域 + 个人号 + 够活跃 + 粉丝量级合理）才进候选。
   - **截断**：大号关注几百个 → 按"和种子/领域匹配度"排序，取 top。
2. **内容指纹重排（辅）**：种子若采过笔记，用其领域/标签/高频选题做指纹，给候选"越像种子越靠前"，并反哺 S2 扩词。

> 抖音：有自己的用户搜索；关注列表/相似能力未逐一确认。**MVP 以小红书为准**，抖音先用 A/B 两路，C 路待确认接口后补。

## 六、候选的"理由"与排序（客观、便宜）

- 每候选给**客观理由**（不逐个调 LLM）：命中方向标签 + 粉丝数 + 发帖量 + 活跃度(最近X天更新) + 火爆度(粉丝+互动,规则算) + 个人号/机构号。
  - 例：`命中「储蓄险科普」· 1.2万粉 · 86篇 · 近5天更新 · 互动率~3% · 个人号`
  - 可选增强：S3 用**一次批量 LLM**给所有候选打"匹配度"标签 + 一句话（控成本）。
- 规则综合分排序：火爆度 + 命中权重 + 活跃 + 个人号优先 +（有种子时）越像种子越靠前。

## 七、循环边界（防打转、防卡死）

1. **种子托盘**：可增删，**上限 3**；加第 4 个提示先移除。每轮用托盘当下的种子。
2. **去重记忆**：会话维护"已展示/已入选/已在库"集合，每轮只给新候选。
3. **挖干刹车**：某轮新候选 < 3（或 0）→ S4 自动提示"这批挖差不多了，换/加领域 or 换种子 or 去结账"。
4. **轮数/预算上限**：单会话 ~10 轮兜底 + 每节点 LLM 预算封顶（防跑飞烧钱）。
5. **出口阶梯（"一直找不到"）**：S4 永远有非循环出口——解释为什么没货（领域太窄/宽、被筛掉、平台本就少）→ 改领域 / 换清种子 / 放宽筛选 / 翻深 / **精确搜索兜底**（知道名字直接去入口①）/ 诚实说"这赛道没几个"。

## 八、成本

每轮：S2 扩词 1 次 LLM（仅首轮/改方向）+ S3 可选批量匹配 1 次 + S4 选项生成 1 次 + N 次搜索/关注列表调用。比旧版"每候选 1 次 LLM"省一个量级；评分（贵）挪到对标库按需触发。

## 九、数据模型 / 状态

新增 **`BenchmarkDiscoverySession`** 持久化一次会话（**多步可迭代向导，刷新/切走不丢**）：
- intent（领域、初始种子）、directions（扩展方向+权重+选中）、seeds（种子托盘）、rounds（每轮候选）、seen（已展示集合）、basket（入选篮子）、stage（当前节点）、status。
- 入库沿用现有"采用 → 创建 BloggerProfile(account_type=benchmark)"。

## 十一、分阶段

- **Phase 1（先落地）**：两入口、删链接评分、评分搬对标库；泛搜索状态机 **S1→S2→S3→S4→S5** 跑通；召回先 **A+B 两路**（用户搜索 + 笔记取作者）；种子用"内容指纹重排"近似；会话持久化。
- **Phase 2（把"找相似"做真）**：**C 路关注列表挖掘** + 种子托盘多轮 + 挖干/出口阶梯打磨；抖音补 C 路（若接口支持）。

## 十二、验证

- 后端单测：多领域扩词；三路召回合并去重；规则综合分排序；关注列表筛子（杂质过滤+截断）；挖干判定；会话去重不重复给；状态机转场。
- 真机：填"香港保险" → 泛搜索能否把"娜姐类"人名号捞出来（重点验 B 路）；设娜姐为种子 → C 路关注列表带出更多同类；多轮不重复；找不到时 S4 出口可用；精确搜索按名字直达；刷新不丢会话。
- 按 CLAUDE.md：ruff + pytest + build → 提交(无 co-author) + push → 部署 → 真机走查。

## 十三、不做的事

- 不做"每候选一次 LLM"的重打分（评分挪到对标库按需）。
- 不让 AI 掌舵流程走向（流程图写死，AI 只填节点内容）。
- 不用"首页推荐"（不针对领域）。
- 不在"找对标"里做账号诊断（那是对标诊断）。
- 抖音关注列表挖掘待接口确认前不强上。
