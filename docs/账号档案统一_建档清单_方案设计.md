# 账号档案统一 + 建档状态清单 · 方案设计

> 目标:把散乱的「采集 / 建档 / 更新」入口,收敛成 **一份账号档案 + 一张建档状态清单**;
> 每行就地弹卡配置、就地显示进度,按逻辑线 gate 用户行为;退役自由「数据采集」tab;
> `mine` / `benchmark` 两类账号同构为一份档案(双方都能蒸馏创作画像)。
>
> 背景见「数据→产出逻辑线」讨论;下游对标分析优化见 `问题清单.md #22`(本次的前置,单独实施)。

---

## 一、为什么(现状的乱)

1. 采用博主后档案是空的(仅昵称/粉丝/头像),要不要建、建到哪、缺什么——**不可见**。
2. 采集有两个入口(自由「数据采集」tab / 档案「一键构建」),做同一件事,还要左边切 tab。
3. 建档任务中断后是**黑盒**:数据其实已部分落库,但用户看不到建到哪、也不知怎么续。
4. 「我的账号」和「对标博主」其实是**同一种实体**,却各有一套管理,概念重复。

## 二、核心模型:账号 = 一份档案(角色参数化)

- 一个 `BloggerProfile`(不论 `account_type` = `mine` / `benchmark`)**就是一份档案**。角色只是标签:
  - `benchmark` = 我要**学**的号 · `mine` = 我要**运营**的号。
- 档案结构两者**完全一致**(现有):资料 · 笔记池 · 数据面板 · 成长轨迹 · 受众需求 · 运营习惯 · 合规体检 · **创作画像** · 笔记池明细。
- **创作画像双方都能蒸馏**:benchmark 的画像 = 可学的打法;`mine` 的画像 = 我现在的打法。
  → 这让对标分析的「我 vs TA 差距」可升级为「两份创作画像逐条 diff」(选题/标题/结构/语言DNA/CTA),留给 #22。
- 前端:`mine` / `benchmark` **共用同一个档案组件**,按 `account_type` 参数化(数据源、少量文案)。

## 三、建档状态清单(核心新增)

每份档案顶部一张清单,对应 `build_dossier` 的阶段。每行:**数据项 · 状态 · 就地入口 · 就地进度**。
判定字段全部来自现有 `dossier_overview`(`pool.total` / `note_total` / `full_count` / `list_count` / `synced_at` / `reached_end` / `portraits` / `building`),**后端基本不用为"读状态"改动**。

| # | 数据项 | 状态显示(判定) | 就地入口(点开弹卡) |
|---|---|---|---|
| ① | 账号资料 | ✓ 粉丝/平台总篇(有 `follower`) | 「刷新资料」(空态「开始构建」) |
| ② | 笔记池 | 已入池 `total`/`note_total`(覆盖率)· 是否到底 | 「增量补采」/「全量校准」 |
| ③ | 数据·轨迹·运营·合规 | 有池即 ✓(`total>0`,读取现算) | 无(自动);受众需求仍按钮触发 |
| ④ | 详情级笔记 | `full_count` / 目标 N | 「升级详情」 |
| ⑤ | 创作画像 | 已蒸馏 ✓(+`stale`/`new_posts_since` 过时标记)/ 未蒸馏 ✗ | 「蒸馏画像」/「更新画像」/「彻底重建」 |

**行为控制(gating)= 按逻辑线卡住**:每行只有**前置就绪**才可点——没池→④⑤灰置;详情 < `distill_min_samples`(8)→⑤灰置。
**一次一操作**:后端对同一博主有构建锁(`build_task_id` + `ensure_no_running_build`),同一时刻只有一行在跑,其余显示「等待中」——正好防并发采集打架。

## 四、就地卡片 + 就地进度(不跳 tab)

交互:**点入口 → 弹 scoped 配置卡 → 确认 → 卡收起、该行原位显示进度条 + 进度文案 → 完成刷新该行**。
进度技术现成:后端任务发 `record_task_event` 进度、`dossier_overview.building` 带 `task_id/status/message`、前端有 `LiveProgress`。主要是前端组织。

各卡内容(**scoped,不把整个自由采集搬进来**):

- **① 资料**:无表单,确认即 `refresh profile`。
- **② 笔记池**:二选一「增量补采 / 全量校准」→ `pool/sync`(mode)。
- **④ 升详情 / 采集内容**(唯一有配置的采集卡,**两种模式二选一**):
  - **模式 A · 批量升**:系统选片(见第五节)。旋钮:**采几篇**(数字,默认 `dossier_default_full_count`)+ **图文 / 视频** 筛选 + 底部**成本预估**。ASR 默认开、评论默认 20、最近/高赞排序内部算法——均不显示。→ `run_blogger_collection`(backfill)。
  - **模式 B · 定向补采**:**粘贴笔记链接**(可多条)补/深挖指定笔记(含列表没抓到的)。复用现有 URL 定向采集能力。
  - 两个模式同处一张卡(如卡内 tab 切换),不设独立入口。
- **⑤ 创作画像**:「更新画像」(`redistill`,便宜重蒸)/「彻底重建」(`build_dossier`,重拉+重升+重蒸);无画像时「蒸馏画像」。

**中断可恢复**:任务中断后已完成阶段都已落库(每阶段边跑边 commit),锁在 `finally` 必释放,`building` 转空。清单据此显示"④ 42/90、⑤ 未蒸馏",提供「继续建档」(= 幂等重跑 `build_dossier`,②增量去重/④backfill 跳过已完成)或只补缺的那步。#20 已修:超时/取消现在干净失败,不再孤儿续跑。

## 五、选片算法(升详情"选哪些")

替代现在的「最新 80 + 高赞 10」两批,统一为一套排序 + 爆文保底:

输入:全量**列表级**池(每篇赞藏 + 池内**中位数**,建②时已算)。

1. (若卡里选了)先按 **图文 / 视频** 过滤候选。
2. 候选 = 赞藏 **≥ 中位数**(="够高赞")。
3. 排序:**时间 新 → 旧**,取用户输入的 **N** 篇。
4. **爆文保底**:强制纳入**全时段 Top-K 爆文**(不论多久以前,防止老爆文被最近挤掉——蒸馏最想学爆文打法)。
   - **K 随笔记总数动态**:`K = clamp(round(note_total × 0.03), 3, 12)`(可入 config,如 `dossier_hot_ratio/floor/cap`)。
     例:100篇→3 · 300篇→9 · 400+篇→12(封顶)· 不足 100→3(保底)。
   - 保底爆文计入 N(最终升详情集合大小 = N,其中含 K 篇爆文 + (N−K) 篇最近高赞)。
- 默认建档(自动)N 沿用 `dossier_default_full_count`(80);不足全采。
- 后端改:`build_dossier` 阶段④ + `_sample_post_ids` 之外新增一个"选片"函数(中位数过滤 + 动态 K 保底 + 模态过滤);单测覆盖。

## 六、定向采集(并入 ④ 卡片,非独立)

「粘贴笔记链接补特定笔记」是 ④「升详情 / 采集内容」卡片里的**模式 B**,**不单独设入口**——同一张卡里「批量升 / 定向补」二选一(见第四节)。用于补漏列表没抓到、或想精确深挖的指定笔记,复用现有 URL 定向采集能力。

## 七、数据采集 tab 退役

- 移除左侧「数据采集」tab。能力迁移:全量池同步→②;升详情→④(模式 A);URL 定向→④(模式 B),都在档案清单里。
- 迁移校验:确认没有别处依赖该 tab;`mine` 账号的采集也一律走档案清单。

## 八、导航 / 结构变化

- 「我的账号」管理区 → **「我的档案」**(复用 dossier 组件,`account_type=mine`;实施时确认现有我的账号视图结构再合并)。
- 「博主档案」保持,组件与我的档案同源。
- 退役「数据采集」tab。
- 「诊断我的」(self)/「对标分析」(benchmark)照旧,各自从对应档案读数。

## 十、验证 / 风险

- gating:前置未就绪时对应行灰置、不可误触。
- 一次一操作锁:某行跑时其余「等待中」的 UX 正确。
- `mine` 档案跑通:我的账号能建档 + 能蒸馏创作画像。
- 退役采集 tab 不破坏:我的账号采集、URL 定向采集仍可用。
- 选片算法单测:中位数过滤 + 动态 K 保底 + 图文/视频过滤 + N 不足全采。
- 中断态:半成品清单显示正确、「继续建档」幂等续跑。

## 十一、不做(本轮范围外)

- 对标分析 P1(读档案去重)/ P2(两份画像 diff)—— #22,本设计是其前置,单独实施。
- 不改蒸馏内核 / 创作链路。
- 不动早报(微信)独立产线。
