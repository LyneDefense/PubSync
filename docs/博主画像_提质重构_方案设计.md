## 0. 主线:一个真相 —— 「账号事实」和「创作画像」分两条线

这轮所有决策都收敛到一条主线:

- **账号事实(事实层)**:身份、数据、运营习惯、成长轨迹、合规体检。
  - **全量笔记池算、一份真相、建档时算、随池同步重算。**
  - 便宜(多数只吃列表数据)、客观、不依赖"用户想学什么"。
- **创作画像(画像层)**:认知 / 人设 / 选题角度 / 分车道写法。
  - **系统选样、深读详情、一份、重蒸即覆盖。**

判断一个东西归哪条线,只问一句:**它变不变取决于"用户意图"?** 不变 → 事实层;变 → 判断层(见第八节对标分析)。

这条线同时解决了一个内在矛盾:**"看博主底色" vs "学他现在怎么做"**——
- 想理解演变/底色 → 看**事实层**(全程)
- 想学当前打法 → 看**画像层**(近期),别跟三年前的风格混在一起

## 1. 档案页整体结构(重构后)

```
基本信息    ← 事实(身份)
数据面板    ← 事实(全量池统计)
运营习惯    ← 事实(全量池行为统计)         【待你最终定,见第四节】
成长轨迹    ← 事实(趋势 + 原因)  ★ 招牌
合规体检    ← 事实(规则扫描)
创作画像    ← 画像(系统选样蒸馏,一份)
笔记池      ← 原料(列表级 + 详情级)
```

去掉的:**用户选笔记、快照、多画像并存**(见第六节)。

## 2. 基本信息

**改动:补「博主主页简介」,且和「我的备注」分开。**

- 现状:`BloggerProfile.description` 存的是**用户填的备注**;平台主页简介(个性签名,如阿璐"没灵感了回来看看阿璐笔记")**没单独存**。
- 做法:新增 `signature` 字段,**刷新资料时从 user_info.signature 落库**,只读展示。
- 边界:`signature`(平台来、只读、随刷新更新) 与 `description`(用户备注、自己写) **两个字段,不混用**。

## 3. 数据面板

| 项 | 决策 |
|---|---|
| **总笔记数** | 已有(`note_total`),补展示即可 |
| **已收录笔记 + 覆盖率** | 新增。展示成一对:`共 288 篇 · 已收录 52 篇（覆盖 18%）`。"已收录"取代内部黑话"入池" |
| **获赞与收藏** | 取**账号级**(user_info.interactions,覆盖全部笔记),**不是**池内求和 |
| **篇均点赞 / 赞藏比** | 保留 |
| **互动率** | **砍掉**。定义是 (赞+藏+评)÷浏览量,而小红书不公开浏览量 → 永远算不出,不留空卡 |
| **图文/视频占比** | 类型在拉列表时逐篇即定(`normal=图文`,其余`=视频`,XHS 无第三类型);按**已收录**算,配覆盖率说明代表性 |

**统一口径:所有统计都按"全量列表池"算,不是按"有详情的那一小撮"算。** 图文/视频占比现在显示 52 篇,就是因为在按详情子集算——这是要修的点。建档必须把**全量列表拉全**(到底/到上限 300),让"已收录 ≈ 总数";覆盖不全就老实标覆盖率,不假装是整号。

## 4. 运营习惯(事实主体 + 蒸馏解读)

**定稿:主体由事实层构建,蒸馏层的解读附上去,并标明来源。**

- **主体 = 建档时算的事实**(全量池、一份真相、随池重算):发布节奏、视频/图文均赞对比、体裁分布(清单体 X/Y)、评论引导占比等客观行为统计。需要内容分析的项(评论引导识别、清单体分类)把分析从蒸馏挪到**建档阶段**(跟合规扫描一个道理)。
- **解读 = 蒸馏层附加**:像"重要内容用口播承载""图文走清单体便于收藏"这类带判断的话,**附在对应事实下方,并加「解读」标注**,让用户分得清哪是算出来的、哪是模型解读的。
- 好处:**数字稳**(不随画像变、不随重蒸变)、**解读活**(蒸了就有);两层分开,底数不打架。

### 4.1 数据正确性:评论引导别错源(评论作者身份)

**发现的坑**:`normalize_comment`(post.py)现在只存 `content / like_count / created_at`,**不记评论作者**,分不清"博主自己的回复"和"别人发的评论"。所以"评论引导 22.6%(如「帮我看看号吗」)"是把**读者的评论**当成了**博主的运营习惯**——错源。

**修正**:
- **评论引导(运营习惯)改从「博主自己的内容」判**:标题/正文/口播里是否主动引导("评论区聊聊""扣1""你怎么看"),不看读者评论。
- **采集评论时记作者身份**:抓 comment 作者 user_id,与博主 user_id(`external_id`)比,打 `is_author`。据此另出一个真运营习惯——**博主回复习惯**(回复率/是否置顶)。
- **读者评论 = 受众信号,不是博主习惯**:"帮我看看号吗"属受众需求,归受众侧(本轮先不做,至少不误挂运营习惯)。
- **蒸馏侧**:喂进去的评论也带作者身份,别把读者的话当成博主的声音。

## 5. 成长轨迹(趋势 + 原因)★ 招牌

这是这轮认定的**档案差异化价值**:告诉用户这个博主的**发展趋势**以及**原因**,而不是给一堆蒸馏变体。

### 5.1 诚实边界(先说不能做的)

- **画不出涨粉曲线**:TikHub 没有历史粉丝端点,只有当前粉丝数。
- **不讲"稳步上升"**:单篇互动是起落的噪声,硬画上升线是骗人。
- **不断言因果**:不说"这篇带来 +1w 粉"。

### 5.2 噪声里扛得住的信号(轨迹的真定义)

轨迹不是一条线,是**分段 + 可描述的形状**:

1. **基线台阶**:看滚动中位数,不看单篇。基本盘 500→3000→稳在 3000,是"地板抬高了"——最抗噪。
2. **突破事件**:远离基线的尖峰(已有 burst 检测)。
3. **波动性/稳定度**:紧贴基线(稳定输出)还是大开大合(碰运气)——"起起落落"本身是信号。
4. **发布节奏随时间变化**(提速/掉线),便宜且稳。

### 5.3 赞藏怎么替涨粉:看"台阶",不看"量"

> 一篇**真涨粉**的笔记留下的指纹:**一个尖峰 + 之后基线被永久抬高**(新粉丝给后面每篇都点赞)。爆了但没涨到粉的,只留孤峰、基线不动。

所以光靠赞藏,也能把**「涨粉型爆文」和「一次性爆文」分开**——看尖峰之后**地板有没有上台阶**。赞藏能替涨粉的,不是量,是**拐点**。

### 5.4 怎么切阶段(核心难点)

**两个坑先躲开**:① 阈值硬切(一换博主就崩);② 把"起步/爆发/平稳/低谷"当预设模板硬塞(很多博主根本没有爆发,或一上来就高位)。

**两条原则**:① 切在哪,数据定(变点检测);② 叫什么名,形状得配得上(不是四选一)。

**流水线**:
1. **按时间分桶**:活跃号按月、低频号按季,每桶取**中位数互动**(抗单篇爆文带偏)。
2. **变点检测**(PELT/二分分段,不用阈值):找均值水平的**持久突变点**。两个硬门槛——**效果量**(真台阶不是小抖动)+ **最短时长**(一个月爆一下不算一个阶段)。
3. **按测出的形状贴标签**,贴不上用中性词:
   - 首段+明显低于后面 → 起步期
   - 相对前段台阶式跳高 → 突破期
   - 高位+低波动+走平 → 平稳/成熟期
   - 从高处台阶式跌下 → 滑坡期
   - 都不贴 → "高位起伏期"这种描述性名。**没有博主被强塞模板。**
4. **退化直接说人话**:无显著突变 → "无明显阶段,一直平稳/起伏";篇数太少 → "样本不够,只给整体,不切阶段"。

**一个统一**:**阶段边界 = 基线持久台阶 = 涨粉型爆文拐点**,是**同一个计算**。切阶段和找涨粉拐点不是两件事。

### 5.5 原因(归因)

找到台阶式爆文后,拿它跟**它自己之前的基线那组**比(对比组,复用爆文归因那套)——选题转了?形式转了?钩子变了? 这才是站得住的归因,不硬编因果。

### 5.6 谁做什么

**数学负责切段(变点检测)→ 规则负责命名(形状→名字)→ LLM 只负责解释"这段为什么起来"。** 每层只被信任它该被信任的那点。

### 5.7 硬约束(说在前面)

- **能切几段取决于篇数×发布密度**,调参救不了。阿璐 30 天/篇 ≈ 一年十几篇,基本切不出可信阶段,顶多"整体高位、低频更新"。
- **最近 1-2 周的笔记还在发酵**,赞藏没攒够,曲线尾巴天然偏低,别误读成掉队。
- 产出**连曲线一起给用户看**,让他自己扫一眼,不甩"处于平稳期"当圣旨。

## 6. 创作画像:去多画像 + 系统选样

### 6.1 决策:彻底去掉「用户选笔记 + 快照 + 多画像」

- 一个博主**一份**创作画像,**重蒸即覆盖**(refresh-in-place)。
- 理由:选"哪些笔记去蒸馏"是把用户做不好的决定甩给他;多画像价值虚;"所有笔记就是他的轨迹",理解底色要看全程;和"一个真相"主线一致。

### 6.2 关键澄清:"不让用户选" ≠ "全部笔记深蒸"

全量 288 篇做详情级蒸馏(每篇正文/转写/图文理解 + 大上下文合成)成本和上下文扛不住。**采样没消失,只是从"用户决定"变成"系统决定",藏起来了。**
- **趋势/事实层**吃全量列表数据(便宜)——"所有笔记=轨迹"在这层字面成立。
- **创作画像层**跑系统选的代表性子集深读。

### 6.3 系统采样规则(与阶段**解耦**)

采样有一条**自己就能站住的基准规则**;阶段只在证据足够强时做一次"减法"。**脆弱的阶段层缺席,采样照跑。**

**基准规则**(有没有阶段都一样,封顶到预算,如详情级 ~35 篇):
1. **近期分层配额**:按时间分档(近 3 月 / 3–12 月 / 更早),越近名额越多——近期加权,不硬切。
2. **表现分层**:每档都要抓到**爆文 + 基本盘**(学天花板也学平时,避免幸存者偏差)。
3. **模态配比**:图文/视频按占比取,各有下限(喂饱分车道)。

**阶段的唯一角色**:当且仅当切出真转型(伴随选题/赛道漂移),把**转型前的笔记从候选剔掉**;剔完不够就放宽退回基准规则。

**退化自洽**:
- 无阶段(数据够)= 整段是一个连贯时代 = **全池采样**(正确,不是兜底失败)。
- 太稀疏 = 样本≈全部,**有多少拿多少**,"选"几乎不存在。

**特点**:确定性、只用数字(赞/时间/类型 + 阶段边界)、不掺 LLM、可复跑;池子一同步重蒸就用同规则重采,画像永远贴当前;**样本如实公示**("基于 35/52 篇:近期加权,含爆文与常规,图文/视频兼顾")。

**残留(诚实)**:真转型但太稀疏没检测到(漏报)→ 近期加权已把权重压在当前人设、伤害有限;要更稳可加便宜的"选题/标签漂移探测"当兜底,不用上完整变点。

### 6.4 不受影响

**lane-split(图文/视频分车道)照旧**——那是一份画像内部的分层,不是多份画像。

### 6.5 创作画像的内容结构(选题思路单列)

一份创作画像按以下块呈现:
1. **一句话看懂**(one_glance)
2. **认知层**:核心信念 / 观点张力 / 价值立场
3. **人设声音**:自称 / 语气 / 口头禅
4. **选题思路** ★ 单独成块(原埋在 `angle_layer`,提出来放大):
   - **选题方法/逻辑**(主):他**怎么决定做什么**——找角度的思路、切入视角、选题复用套路。这是**可迁移到你自己赛道的方法**,是这块最值钱的部分。
   - **常用角度**(topic_angles)、**借势/蹭热点方式**(trend_hijacking)。
   - **证据**:选题聚类(事实层从标题/标签聚出来的"他都做哪些题")挂下面当佐证。
   - 口径:选题方法是**解读**(模型推断、画像层);底下的选题聚类是**事实**——沿用"事实 + 解读分层"。
5. **分车道写法**(lanes):标题公式 / 开头 / 封面文案 / 图内编排 / 语言 DNA。

**为什么单列**:选题是最高杠杆的创作决策,对"学一个博主"的人,"他怎么持续选到能爆的题"往往比标题公式更想偷;而且它正是**成长轨迹里爆发拐点最常见的原因**——创作画像和轨迹在这里交汇。

## 7. 合规进画像(合规体检)

**决策:博主合规扫描是纯"账号事实",搬进档案。**

- **为什么是事实**:`scan_account` 的规则包由博主**自己的 niche/tags/标题**自动识别(`detect_verticals`),命中和评分只依赖博主自己的笔记文本。全链无一环依赖"用户想学什么"。
- **为什么值得放**:它是套在**模仿链路**上的护栏——你要从这个博主蒸馏 skill 照着做,若他靠极限词/绝对化/虚假承诺,蒸出的 skill 会继承这些风险。档案上先亮红灯,甚至把信号喂进蒸馏("此号写法有合规风险,注意规避")。
- **怎么落**:建档长任务加一个**廉价规则扫描阶段**(P1 不花钱、不调 TikHub、不用大模型),池子同步后自动重算。P2 的 LLM 裁决(有成本)留在对标分析按需触发,不进档案。
- **覆盖度诚实**:标题级扫描覆盖全量池(标题里的极限词最常见);正文级只覆盖详情级笔记——跟数据面板"详情级 N 篇"一个口径。

## 8. 对标分析 = 画像的评分层

**决策:对标分析从"自采自算"改成"给画像打分"。**

- 现状问题:它自己临采 30 篇最新 + 自己重算硬实力(触达/藏赞比/爆文/活跃),而这四维**画像已经算了、且更准**(全量池 vs 30 篇)。带来:①一个号两套数字 ②白花 TikHub ③软实力只拿标题喂模型。
- 改造后:
  - **硬实力** ← 读画像(数据面板 + 轨迹),分档打分。零额外采集,和画像页永远一致。
  - **合规** ← 读画像(合规体检);用户填了品类就在博主基线上**叠加**一次扫描。
  - **软实力 + 契合意图** ← 唯一还需当场判断的、真正"针对你意图"的部分,且喂**详情级**料(正文/转写/图文),不再只喂标题。
  - **值不值得对标的结论 + 报告 + 历史**。
- **意图引导 quiz 保留**(那是关于用户目标的,不属于博主事实);其"读取 TA 最近 N 篇"改读池子,不再重拉。
- **self(诊断我的账号)分叉**:我的账号不一定建了画像 → 保留临采兜底,有池子时优先用池子。

## 9. 要拆掉 / 改动的东西(scope)

- **删**:快照表、选笔记 UI、多画像并存 UI(`DossierPortraits` 多行 + 重新蒸馏)。
- **改**:蒸馏入参从"snapshot 的 post_ids"→"系统选样";对标分析硬实力/合规改读画像;`normalize_comment` 记评论作者身份(`is_author`);评论引导改从博主自身内容判。
- **合规体检 UI**:加一行小字"依据已收录的 N 篇笔记"(覆盖度)。
- **加**:`signature` 字段;账号级获赞收藏解析;数据面板全量列表池口径 + 覆盖率;运营习惯事实模块(事实主体 + 蒸馏解读附加 + 说明);成长轨迹变点分段 + 台阶/涨粉拐点 + 阶段级归因;合规建档阶段;创作画像「选题思路」单独成块。
- **不动**:lane-split。

## 10. 拍板结果(已定)

1. **运营习惯**:✅ 事实主体 + 蒸馏解读附加(加「解读」说明)。
2. **去多画像**:✅ 彻底去掉"用户选笔记 + 快照 + 多画像"。
3. **实施**:✅ 全部实施。
4. **选题思路**:✅ 创作画像里单独成块(方法为主 + 事实聚类佐证)。

## 11. 明确不做

- 涨粉曲线、"稳步上升"叙事、因果证明。
- 全量笔记深蒸(改系统选样)。
- 费用监控在用户端展示(留后台,后面做)。
- 硬删笔记(只软删/下架)。
