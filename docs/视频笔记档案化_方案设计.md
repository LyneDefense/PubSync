# 视频笔记档案化 + 模态收敛 · 方案设计

> 状态:方案设计(完整版,未写代码)
> 日期:2026-07-08
> 一句话:**把「口播/非口播」从采集时拍死的门,翻转成从视频详情派生的标签;每条视频采集时把能白拿的数据整理成「视频档案」存进笔记详情,贵的深层数据在建档时补;蒸馏车道从 图文/口播/非口播 三条收敛成 图文/视频 两层。**
> 取代:`视频制作手法蒸馏_方案设计.md`(其 L1/L2 抽取被吸收进本方案第 6 节)。

## 决策(已定,2026-07-08)

1. ✅ **P0 就这么定**:模态收敛 + L0 档案 + 派生标签 + 删 T2 + 详情展示。
2. ✅ **视频层收敛成一条**(不保留 talking/visual 两条)。
3. ✅ **L1/L2 在「采集这条详情」时触发**(不是建档时)——理由:采集时视频已为 ASR 下载、且**视频直链新鲜**(小红书直链会过期,拖到建档要重下或失效)。**蒸馏时若发现某视频只有语音/字幕稿、缺镜头/运镜(只到 L0)→ 显式提醒**(并可"补采视频画面")。
4. ✅ **先小红书**,抖音同管线后续再接。

---

## 一、动机(为什么翻转)

现状(代码级):

- 模态细分 `content_subtype`(image_text / talking_video / visual_video / unknown)是**采集时**由 `classify_subtype`([modality.py:42](../backend/app/blogger_distillation/modality.py#L42))用薄信号拍死的:平台 → 字/秒密度 → 模糊带再走一次 **T2 LLM 裁决**([collection.py:643](../backend/app/blogger_distillation/service/collection.py#L643))。
- 蒸馏内容层直接按这个门分 3 车道([distillation.py:169](../backend/app/blogger_distillation/service/distillation.py#L169))。
- 一条视频只抽 `ASR 逐字稿 + 封面一张图`([note_pipeline.py:120](../backend/app/blogger_distillation/service/note_pipeline.py#L120));**制作层(分镜/节奏/运镜)一个字段没有**。

三个毛病:① 门是早早拍死的,薄信号(字/秒)易误判,还为此多花一次 T2 LLM;② 非口播车道"薄",是因为除了逐字稿没别的可学;③ `content_subtype` 是硬 label,想换 taxonomy 要重采。

**翻转后**:模态只认 图文/视频(= 已有的 `content_type`,平台给的硬事实);口播/非口播这类**从采集到的「视频档案」里事后派生**——可调、可回溯、不重采;蒸馏不再被早 label 硬分叉。

---

## 二、核心决策

1. **模态 = 图文 / 视频**(`content_type`)。`content_subtype` 不再当"门"。
2. **口播/非口播 → 派生标签**:从每条视频的 `video_profile`(见第 4 节)用纯函数算出来,存下来给人看/筛选/给蒸馏当提示,但**不门控管线**。
3. **蒸馏内容层:3 车道 → 2 层(图文 / 视频)**。视频层是**一条**车道,prompt 同时吃逐字稿(话术)+ 视频档案(拍法),按数据自动加权。口播视频天然话术重、视觉流天然拍法重——权重来自数据,不用硬门。
4. **顺带删掉 T2 LLM 裁决**:不再需要采集时把视频拍成 talking/visual 的精确 label,省一次 LLM 调用。
5. **成本分层采集**:能白拿的(L0)采集时全量做;贵的(L1/L2)建档/蒸馏时补。

> 为什么收敛成一条视频层比保留 talking/visual 两条更好:博主的视频风格通常一致(口播号大多口播,视觉流号大多混剪);两车道是"数据薄"时代的补丁。有了每条视频的富档案,一条视频层 + 富证据 > 两条薄车道,且天然兜住"混合型"博主。若某博主两种都做得多,交给 LLM 在视频层里分小节,而不是管线硬分叉。

---

## 三、目标产物(用户/系统拿到什么)

- **笔记详情页**:每条视频渲染「视频档案」——L0 永远有(逐字稿、时长、口播浓度、封面解析、互动);L1/L2 建档后有(镜头数、节奏、分镜表、出镜)。这本身是个有用的新功能。
- **派生标签**:口播浓度(无/低/中/高)、出镜(是/否)、节奏(慢/中/快),或汇总一个粗档"视频类型"。展示 + 筛选。
- **蒸馏产物**:视频层画像 = 话术打法(选题/钩子/脚本结构)+ 拍法打法(分镜结构/节奏/封面公式)。
- **创作产物**:视频类创作从「口播脚本」升级为「**分镜脚本(shot list)+ 剪辑节奏建议 + 封面方案**」(口播浓度高的偏脚本、视觉流偏分镜)。

## 四、数据结构:`video_profile`(分层 JSON,落 BloggerPost)

一列 `video_profile: Text`(JSON),分层填充;派生标签单独一列便于索引/筛选。

```jsonc
{
  "layer": "L0|L1|L2",              // 已填到哪层
  // —— L0(采集时,白拿)——
  "duration_s": 182,
  "transcript_len": 640,
  "speech_cps": 3.5,                // 字/秒(已有逻辑)
  "narration_level": "high",        // 派生:无/低/中/高
  "cover": { ...visual_digest 已有... },
  // —— L1(建档时,CPU 镜头切分)——
  "shot_count": 42,
  "avg_shot_s": 2.1,
  "cuts_per_min": 28,
  "pace": "fast",                   // 派生:慢/中/快
  // —— L2(建档时,VLM 代表帧)——
  "on_camera": true,                // 派生:是/否/半
  "hook_3s": "怼脸提问 + 大字标题",
  "transitions": "硬切为主",
  "shots": [ {"t":"0-2s","scale":"近景","on_camera":true,"caption":"人均30吃到扶墙","desc":"出镜抛问题"} ],
  "style_summary": "快节奏出镜口播 vlog,靠花字和卡点维持信息密度"
}
```

派生标签(单独列或小 JSON `video_tags`):`{narration_level, on_camera, pace, video_kind}`。

## 五、分层采集(成本分层 —— 关键取舍)

**"采集时把数据都整理好"不能对全量视频做到底**:深层要下载+解码+VLM。按成本分层:

| 层 | 内容 | 成本 | 何时采 | 依赖 |
|---|---|---|---|---|
| **L0** | 时长、逐字稿长度、口播浓度、封面解析、互动 | **已有,不加钱** | 采集时,全量 | 无(全用现成字段) |
| **L1** | 镜头数、节奏(cuts/min)、时长分布 | 便宜(CPU) | **采集这条详情时**(复用 ASR 那次下载) | ffmpeg 解码本地文件 |
| **L2** | 逐镜头画面、出镜、分镜表、hook | 中(VLM 代表帧,~¥0.03–0.05/条) | **采集这条详情时** | 镜头切分 + 多图 VLM |

**触发点 = 采集详情时**(决策 3):此刻视频已为 ASR 下载、直链新鲜,L1/L2 直接复用,不必等建档重下(小红书直链会过期)。范围**只覆盖真正被详情采集的新笔记**(= 用户选的 `sample_limit`/`amount`,本就受控),不是全量 300 候选池;`note_key` 命中已处理过的直接复用(照 ASR/vision reuse),重采不重复付费。可用后台开关关掉 L2(留 L1);关了则该视频停在 L0,蒸馏时按下面「缺详情提醒」处理。

**L1/L2 抽取细节**(即原「制作手法」方案):镜头切分用 [PySceneDetect](https://github.com/Breakthrough/PySceneDetect)(CPU 出精确节奏,不花 VLM 钱)+ 每镜头 1 张代表帧跑 VLM(自适应、封顶 ~15–20 帧)。**不用均匀抽帧**(3 分钟快剪太稀、测不出节奏),**也不直接喂视频模式**(按 ~1fps ~180 帧、贵 3–6× 且 LLM 数不准镜头)。心智模型:**帧数 ≈ 价格**,所以节奏交 CPU、风格只给少量帧。GLM-4V-Plus ¥4/百万 token([文档](https://docs.bigmodel.cn/cn/guide/models/vlm/glm-4v-plus-0111),图像/视频每帧 token 未公开,需实测)。

**下载共享**:视频已为 ASR 下过一次,但下载在 `asr_provider.transcribe_video_url()` 内部、下完即弃([asr_step.py:123](../backend/app/blogger_distillation/service/asr_step.py#L123))。L1 需要解码 → 把「下载一次」提到管线层,ASR 和抽帧共享本地文件(一个不大的重构),避免重复流量。

## 六、派生标签:从档案算,不在采集时猜死

一个纯函数 `derive_video_tags(video_profile) -> video_tags`,规则化、可调、可回溯:

- `narration_level`:L0 的 speech_cps → 无(无转写)/低/中/高。
- `pace`:L1 的 cuts_per_min → 慢/中/快(没 L1 则 null)。
- `on_camera`:L2 判定(没 L2 则 unknown)。
- `video_kind`(可选粗档):综合上面给一个人话标签。

标签随层数变丰富:L0 只能给口播浓度,L1 加节奏,L2 加出镜。**换 taxonomy 只改这个函数,不动管线、不重采。**

## 七、蒸馏车道收敛(承重墙)

改 [distillation.py:168-202](../backend/app/blogger_distillation/service/distillation.py#L168):

- **内核层**:不变(吃全部笔记,跨模态)。
- **内容层**:`lane_posts` 从 `{image_text, talking_video, visual_video}` 收敛成 `{image_text, video}`——按 `content_type` 分,不按 subtype。
- **视频层 prompt(distill_lane 的 video 分支)升级**:每条视频给 `逐字稿 + video_profile`;prompt 要它同时产**话术**(选题/钩子/脚本,按 narration_level 加权)+ **拍法**(分镜/节奏/封面)。
- **artifacts**([distill_engine](../backend/app/blogger_distillation/service/distill_engine.py) + skill md / HTML 报告):内容层小节从「口播写法 / 非口播写法」改成「图文写法 / 视频写法(话术+拍法)」;保留旧形态兜底。
- 样本不足兜底(现有 dominant 综合逻辑)保留,按新两层适配。
- **缺详情提醒(决策 3)**:蒸馏视频层前,统计视频池里 `video_profile.layer` 只到 L0 的比例(即只有语音/字幕稿、无镜头/运镜)。若占比高 → 蒸馏事件 + 报告里显式提醒「N 条视频只有语音稿、缺画面/节奏数据,拍法分析可能不全」,并给「补采视频画面」入口(对这些笔记重采详情→补 L1/L2;直链已过期,走重取详情/URL 定向采集那套)。存量(P0 回填只到 L0 的老视频)正是这种情况。

## 十一、失败与降级

- 无视频流/抽帧失败/VLM 失败 → 该层留空、`video_profile.layer` 停在能到的层,**不影响整批**(照 #20)。派生标签只用已填层。
- 图文笔记 → 只有 content_type=image,不走 video_profile。
- 视频无转写(ASR 关/失败)→ narration_level=无,L0 仍有时长/封面。
- 低置信描述 → 标「AI 推测」,不当事实 [[dossier-quality-refactor]]。

## 十二、成本 / 收益账

- **L0**:0 新增(全用现成字段)——却能立刻给"视频档案 + 口播浓度标签 + 详情展示"。
- **删 T2 裁决**:**省**掉现有每条模糊视频一次 LLM。
- **L1**:CPU,几乎 0 token;**L2**:~¥0.03–0.05/条,采集详情时随下载做掉、`note_key` 复用不重付,可后台关(关了停在 L0、蒸馏提醒)。
- 净效果:日常采集省一次 T2 裁决;拍法数据在"已经下载"的那一刻顺手做掉,不额外重下。
