# 蒸馏证据供给优化 · 方案设计

> 关联：[[蒸馏车道化_方案]]（本方案改的是车道化蒸馏的"喂料层"）、[[视觉层_图片理解_方案设计]]（图文数据来源）。
> 本文只聊"喂给蒸馏 LLM 的证据怎么组织"，不动采集、不动视觉层、不动车道划分。

## 背景与坐实（为什么改）

我们上线了图文理解（`image_text` 逐张文字 + `visual_digest` 封面钩子/版式/风格/逐张说明），期望 skill 蒸馏因此更准更细。但实测发现"数据加了、skill 没变好"。用阿璐真实样本（50 篇）量化 `analysis.analyze_posts` 产出的 stats 喂进 prompt 前后的损耗，坐实了根因：

| 指标 | 内核·全账号 | 图文车道 |
|---|---|---|
| stats 序列化后总长 | **89227 字符** | 88882 字符 |
| prompt 硬截断上限 | 16000 | 16000 |
| **被丢弃比例** | **82%** | 82% |
| `representative_posts`（20 条代表样本）段起始偏移 | **31519** → 整段在窗口外，0 条进模型 | 26985 → 同上 |
| 图文样本图内文字进模型 | **18 条只进 4 条** | 20 条只进 4 条 |
| 图内文字字符进模型比例 | **仅 20%** | 仅 21% |
| `opinion_sentences`（含图内金句）段起始偏移 | **83821** → 完全没进 | 83411 → 同上 |
| hot_posts ∩ representative | 10/10 全重叠（同一大 payload 塞两遍） | 8/8 全重叠 |

**结论**：现在的做法是 `json.dumps(stats, ...)[:16000]`——把一个 8 万多字符的大字典整包序列化后盲砍前 16000。模型实际只读到前 18%（大致到 hot_posts 中段），**20 条代表样本一条都没读到、图内文字只进 20%、认知层的观点金句池完全没喂**。这不是"图文被稀释"，是整条证据供给一直在挨饿、蒸馏一直半盲工作。

## 目标

让"该进模型的证据"确定进模型，在**不显著增加 token/耗时**的前提下：
1. 图内文字 + 视觉摘要真正到达蒸馏 LLM（尤其图文车道）。
2. 代表样本、爆款、观点金句池按优先级进入，不再被盲截。
3. 消除 hot/representative 的重复 payload。
4. 输出层（skill/报告）显性承载图文视觉打法，让增量"看得见"。

## 一、根因拆解

1. **整包 dumps + 盲截（主因）**：`build_core_prompt` / `build_lane_prompt` 里 `json.dumps(ctx.stats)[:16000]`（`distill_engine.py:84`、`:155`）。字典序列化顺序 ≠ 信息重要性顺序，尾部（代表样本、金句池）先被砍。
2. **单条样本内部字段顺序不利**：`post_summary`（`analysis.py:351`）里 `image_text_excerpt` / `visual_digest` 排在 `body_excerpt` / `transcript_excerpt` 之后，即便该条进了窗口，图文部分也常被砍。
3. **认知层缺料**：图文号正文往往很短，博主的观点/立场/金句在图里。`opinion_sentences` 已纳入 `image_text`（`analysis.py:342`），但它在 stats 末尾，core prompt 根本读不到 → 认知层闭眼蒸。
4. **冗余挤占**：`hot_posts ⊂ representative_posts`，前部完全重叠，同样的大块 JSON 出现两遍，进一步吃掉本就不够的窗口。

## 二、方案总览（分三层，可分批落）

| 层 | 内容 | 独立性 | 收益 |
|---|---|---|---|
| **P0** | 用"结构化证据装配"替换整包 dumps 截断：紧凑证据块 + 优先级填充 + 去冗余 + **物料层去噪** | 独立、收益最大 | 图文/文本证据确定进模型且去噪 |
| **P1** | 观点金句池（含图内）前置进证据区 | 与 P0 同层，几乎免费 | 认知层不再缺料 |
| **P2** | 输出层补"图文视觉打法"小节（封面文案公式/图内信息编排/版式套路） | 稍独立（改 artifacts + 可能加字段） | 增量在产出里显性可见 |

## 三、详细设计

### P0-1 新增证据装配器 `evidence.py`

新建 `app/blogger_distillation/evidence.py`，单一职责：**把 `analysis` 产出的 stats 字典渲染成"给 LLM 的证据文本"**（预算内、优先级、结构化、视觉信号前置）。吃 dict、不碰 ORM、不碰 DB，纯函数好测。

核心函数：

```python
def render_stats_digest(stats: dict, *, char_budget: int, scope: str = "core") -> str:
    """把 stats 渲染成给蒸馏 prompt 的证据文本。按优先级分段填充，只在尾部兜底截断。
    scope="core"（认知，重金句/观点）/ "lane"（内容层，重爆款证据块）微调段落顺序与配额。
    """
```

分段与优先级（core 与 lane 略有侧重，用 scope 区分）：

| 顺序 | 段 | 内容 | 预算占比（约） | 备注 |
|---|---|---|---|---|
| 1 | 账号统计摘要 | 均赞藏、模态对比、标题/开头/CTA 模式占比、发布节奏、成长趋势、高频标签 | ~20% | 现有 stats 的统计字段紧凑成行，不 dumps 大数组 |
| 2 | **观点/金句池**（P1） | `opinion_sentences`（含图内金句），去重取 TOP N | core ~20% / lane ~8% | **前置**，core 侧重 |
| 3 | **爆款证据块** | `hot_posts` 每条一个结构化证据块（见 P0-2 格式） | ~35% | 完整、视觉信号显眼 |
| 4 | 代表样本精简行 | `representative_posts` **排除已在 hot 的**，每条一行：`标题｜赞藏｜形态｜一句关键信号` | ~20% | 去冗余后的补充广度 |
| 5 | 评论洞察 | `comment_insights_source` 采样若干 | ~5% | 尾部，超预算先砍这里 |

只有第 4、5 段是"可截断"的；1/2/3 段保证进窗口（若单账号样本极多导致 3 段超配额，则 hot 证据块内部按 score 截条数，而非砍字段）。

### P0-2 单条证据块格式

按模态分型，把视觉信号提到显眼位置。示例：

**图文块**：
```
【爆款·赞2.3w藏1.1w评218】变化才是博主最小单位的精神丰容
  封面钩子：先照顾好自己，再谈丰容
  版式：卡片清单｜风格：低饱和莫兰迪、大字标题
  图内要点：
    ①(封面) 变化才是博主最小单位的精神丰容
    ②(清单) 3 个信号说明你该换赛道了：…
    ③(教程步骤) 第一步 先…
  正文摘要：<body_excerpt 前 ~200 字>
```

**口播视频块**：
```
【爆款·赞5.1w】为什么她能靠记流水账plog做到10w+
  口播摘要：<transcript 前 ~300 字>
```

**非口播视频块**：标题 + 封面钩子（若有）+ 一句 desc；诚实标注"动态视觉本次未覆盖"。

要点：
- 图内要点**逐张**列 `role: text`（复用 `visual_digest.images`），去噪后的 text 已经只含内容性文字。
- 每块设**单块字符上限**（如 ~600 字），防单条爆款的超长图内文字吃光预算。
- 来源标注（external_id / 标题）保留，供爆款拆解的 `source` 字段落地。

装配单块复用/对齐 `post_content.visual_context`（已有封面/版式/风格/逐张说明的拼装逻辑），避免两套拼法；`evidence.py` 负责"选哪些、排序、预算、分型标签"，`post_content` 负责"一条的视觉块长什么样"。

### P0-3 去冗余

`render_stats_digest` 内：代表样本段用 `external_id ∉ hot_ids` 过滤，只输出精简行。省下的预算并入爆款证据块 / 金句池。

### P0-4 接入 `distill_engine`

`build_core_prompt` / `build_lane_prompt` 里，把

```python
{json.dumps(ctx.stats, ensure_ascii=False, default=str)[:16000]}
```

换成

```python
{render_stats_digest(ctx.stats, char_budget=settings.distill_evidence_char_budget, scope="core" | "lane")}
```

prompt 里那几句"充分利用 visual_digest / image_text_excerpt"改成对着**证据块结构**说（"每条爆款证据块给了封面钩子/图内要点/正文摘要，据此提炼封面文案公式与图内信息编排"），因为现在证据确实在了。

### P0-5 物料层去噪（四个抓手）

图内文字的噪音分两层，在两处分别治理，不是一个地方：

- **源头层（采集时，已上线）**：VISION_PROMPT 已改为"只抄作者主动放给读者看的内容性文字，跳过背景报纸/水印/路人/包装偶然出现的字"。第一道闸，但 VLM 不完美，仍可能误抄/漏抄。
- **物料层（蒸馏喂料时，本方案补强）**：即便源头有残留噪音，装配证据块时用四个现成的结构化抓手再滤一道——

1. **role 过滤**：图内要点只取**内容性 role**（封面/标题/清单/教程步骤/对比/数据/金句/要点），`role=实拍/截图/路人/商品` 这类低内容密度的降权或跳过。比"逐字全塞"精准得多。
2. **desc 优先、text 兜底**：每张图的 `desc` 是 VLM 已提炼过的"想传达什么（忽略背景杂字）"，抗噪。"这张图讲什么"用 desc，"逐字证据"才用 text 且限长。
3. **单块限长**：每条证据块 ~600 字上限，防单条多图爆款的图内文字爆表、喧宾夺主。
4. **跨样本共性天然稀释单点噪音**：蒸馏是"从 N 条样本找共性"，不信单条；个别图的误抄在 20 条样本里抓规律时被自然滤掉。⚠️**例外**：系统性噪音（每篇都抄了同样的背景水印/固定引导语）不会被稀释，须靠 role 过滤 + 跨样本去重兜住。

外加一句 **prompt 锚**：明示"图内文字以内容性要点为准；装饰/引导字（点赞收藏关注、水印）不要当方法论"，给模型一个去噪指令锚。

落点：role 过滤 / desc 优先 / 单块限长 在 `evidence.py` 的 `_evidence_block` 里；跨样本去重在 `_opinion_pool` 与代表样本段里；prompt 锚在 `distill_engine` 的 core/lane 硬边界里补一句。

### P1 金句池前置

即 P0-1 的第 2 段。`opinion_sentences` 已含 `image_text`，只需：去重（同句/子串）、按含"我觉得/其实/关键/本质…"信号词加权取 TOP N、放到证据区靠前。core prompt 尤其受益（认知层四子项有据可依）。

### P2 输出层"图文视觉打法"小节

1. **车道 schema 补字段**（`distill_engine.build_lane_prompt` 的图文车道输出）：新增 `visual_layout_patterns`（图内信息如何分屏/分卡编排、版式套路），与已有 `cover_text_rules`（封面文案）配对。非图文车道留空数组。
2. **归一**：`_LANE_LIST_KEYS` 加 `visual_layout_patterns`。
3. **渲染**：`artifacts.CONTENT_LABELS` 加 `"visual_layout_patterns": "图内信息编排 / 版式"`，skill markdown 与报告 HTML 自动多一节（`content_section_markdown` / `render_content_section` 已按 CONTENT_LABELS 遍历，无需改渲染逻辑）。
4. 效果：图文号的 skill 里显性出现"封面文案公式 + 图内信息编排 + 版式套路"，用户直接可抄。

## 四、数据结构与接口

- 新文件 `app/blogger_distillation/evidence.py`：
  - `render_stats_digest(stats, *, char_budget, scope) -> str`（对外）
  - `_evidence_block(sample: dict) -> str`（单条，按 `content_type`/`content_subtype` 分型）
  - `_account_summary(stats) -> str`、`_opinion_pool(stats, limit) -> str`、`_representative_lines(stats, exclude_ids) -> str` 等分段私有函数
- `config.py` 新增 `distill_evidence_char_budget: int = 28000`（core/lane 共用；入 `admin/config_registry` 可调）。放弃"字符 16000 盲截"，但仍以字符为预算单位（不引 token 计数库，字符近似足够；文档注明）。
- `distill_engine.py`：两处 prompt 引用改为 `render_stats_digest(...)`；lane 图文分支 schema/normalize 加 `visual_layout_patterns`。
- `artifacts.py`：`CONTENT_LABELS` 加一项。

不新增 DB 列、不加迁移、不改采集与视觉层。

## 六、验证

**单测**（`test_evidence.py`）：
- 预算内：给定 char_budget，输出不超预算；账号统计+金句+爆款证据块（1/2/3 段）在预算充足时全进。
- 优先级：预算紧张时先砍代表样本行/评论，不砍金句与爆款证据块。
- 去冗余：representative 不重复输出 hot 的样本。
- 分型正确：图文块含图内要点（来自 visual_digest.images）与正文摘要；口播块含 transcript 摘要、不含图内；非口播块诚实标注。
- 金句前置：opinion 段偏移 < 爆款证据块偏移。
- 去噪：`role=实拍/截图` 的图内文字被跳过/降权；desc 缺失时回退 text；单块超长被截到上限内；系统性重复引导语被跨样本去重。

**真机（阿璐，图文号）**：
- 重蒸一次，用坐实用的量测脚本复测：证据文本里图文样本图内文字**进模型比例 → 应接近 100%**、代表样本不再整段丢失、金句池在窗口内。
- 人工对比 skill 前后：图文车道的"封面文案公式 / 图内信息编排"应从空泛变具体；认知层四子项更有据。

**新旧 A/B（合入关口，用证据决定值不值得合入）**：
- 做法：同一批阿璐 50 篇样本，**旧证据供给（整包 dumps 截断）vs 新证据供给（结构化装配）各蒸一次**，把两份 skill 并排对比。
- 实现：`render_stats_digest` 留一个"旁路开关"（config `distill_evidence_legacy` 或函数入参），旧路径保留 `json.dumps(stats)[:16000]`，跑一次新、一次旧，不动其它。
- 判据（人工看）：图文车道的封面文案公式 / 图内信息编排是否从空泛变具体；认知层四子项是否更有据、更贴样本。
- **关口**：若 A/B 没有肉眼可见的改善，本改**不合入**（或只留机制、把预算回退）。不靠"我觉得变好了"。

**流程**（按 CLAUDE.md）：ruff + pytest + build → 提交（无 co-author）+ push → `deploy.sh update-backend` → `deploy.sh logs backend` 确认。

## 七、成本与风险

- **Token/耗时**：预算从"实读 16000"提到 28000，单次蒸馏输入 token 约 +75%；但换来的是"该看的都看到"。GLM 上下文足够；耗时增量可接受（蒸馏非高频）。上限入 config 可回调。
- **注意力稀释**：28000 是权衡值——太大反而稀释注意力。用"结构化证据块 + 前置金句"提升信息密度，抵消长度增长。
- **风险低**：纯喂料层改动，不碰数据模型、不碰采集；证据装配纯函数、易测；出问题可通过调小 char_budget 或回退 prompt 引用快速缓解。

## 八、不做的事

- 不引入 token 计数库（字符预算近似足够）。
- 不改采集 / 视觉层 / 车道划分逻辑。
- 不加 DB 列、不做迁移。
- 不做"多轮检索式喂料 / RAG 选样本"——本轮先把"确定该进的证据装进去"，检索式选样是后续可选项。
